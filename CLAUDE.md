# CLAUDE.md
Telegram-бот записи к психологу с уведомлениями.

## Стек
- Python 3.14, `aiogram` 3.x (**webhook-режим**)
- YDB (serverless) через официальный `ydb` SDK (gRPC)
- Деплой: Yandex Cloud Functions
- Пакетный менеджер/раннер: `uv` — при локальной разработке весь код и инструменты запускать через `uv run`

## Домен
Психолог **один на весь бот**. Таблицы/сущности психолога в БД нет — все его параметры берутся из переменных среды и не редактируются из бота.

Параметры (только из переменных среды, НЕ хардкодить):
- Длительность слота — из env, дефолт **20 мин**. Психолог не настраивает.
- Интервалы доступности — из env (напр. `10:00-14:00,15:00-18:00`). **День недели не учитывается**, только время суток; те же интервалы каждый день. Слоты генерируются внутри интервалов с шагом длительности.
- Таймзона — из env: в ней задаются интервалы доступности и в ней (или в таймзоне пользователя) показывается время.
- Текст welcome-сообщения и текст сообщения о невозможности брони — из env.

Бронирование:
- Пользователь может забронировать **только на сегодня**.
- Если на сегодня свободных слотов не осталось — показать сообщение о невозможности брони (текст из env).

Бронь (запись клиента):
- Самостоятельная персистентная сущность в YDB с **неизменным UUID** — не «слот в статусе занят». Слоты эфемерны (генерируются из интервалов), бронь — нет.
- Хранить **явные `start` и `end` в UTC** на самой брони, а не start + подразумеваемую длительность. При бронировании денормализовать `end`, чтобы позднее изменение длительности слота (env) не «сдвигало» существующие записи.
- Отмена — **переход состояния (soft), никогда не hard-delete**. Строка брони переживает отмену.

Инварианты:
- Слот существует только внутри окна доступности (интервалов из env).
- Бронировать можно только слоты на сегодня.
- Один слот нельзя забронировать дважды — проверка при записи обязательна (учитывать гонки).
- Изменение длительности/интервалов (env) не должно затирать уже забронированные слоты — их судьбу решать явно (за счёт денормализованных `start`/`end` на брони).
- **Время везде хранить в UTC**, показывать пользователю в локальном/заданном часовом поясе. Интервалы доступности задаются в таймзоне из env.
- Состояния слота: свободен / забронирован / отменён.

## Архитектура (serverless-ограничения)
- **Только webhook**, никакого long polling — функция эфемерна.
- Хендлер идемпотентен, не держит фонового состояния между вызовами.
- **Уведомления** — отдельная функция по timer-триггеру YC. Никаких APScheduler / фоновых asyncio-задач: они не переживут завершение вызова.
- Клиент YDB инициализируется на уровне модуля и переиспользуется тёплым инстансом; вручную закрывать не нужно.
- Конфигурация и секреты (токен бота, креды YDB, длительность слота, интервалы доступности, таймзона, тексты сообщений) — только из переменных окружения функции, в код не хардкодить.
- Все переходы состояния брони (создание/отмена/изменение) — через **единый доменный сервисный слой**, не инлайн в aiogram-хендлерах. Хендлеры тонкие; доменная логика тестируется без Telegram.

## Рабочий процесс
Spec-driven (OpenSpec): сначала спека изменения — потом код. Не отступать от активной спеки без её обновления.

## Требования к коду (обязательно)
- **Test-first**: сначала падающий тест, затем реализация. Никакой логики без предшествующего теста. Тесты расположены в директории tests.
- Покрытие ≥ 80%.
- Перед завершением любой задачи обе команды должны быть зелёными:
  - `make test` — тесты
  - `make validate` — линтер + форматтер + тайп-чекер
- Всё исполнять через `uv run` (тесты, скрипты, инструменты).
- Типизировать публичные функции; проходить тайп-чекер без `# type: ignore`-заглушек.

## Не делать
- Не использовать polling.
- Не добавлять планировщики/фоновые задачи внутри функции.
- Не писать код без предшествующего теста.
- Не коммитить креды и токены.
