# CLAUDE.md
Telegram-бот записи к двум психологам с уведомлениями.

## Стек
- Python 3.14, `aiogram` 3.x (**webhook-режим**)
- YDB (serverless) через официальный `ydb` SDK (gRPC)
- Деплой: Yandex Cloud Functions
- Пакетный менеджер/раннер: `uv` — при локальной разработке весь код и инструменты запускать через `uv run`

## Домен
Параметры (НЕ хардкодить):
- Психологи — записи в YDB (сейчас двое, но количество меняется). Никаких enum/констант на число специалистов.
- Длительность слота — **редактируемый атрибут психолога** (каждый настраивает сам, сейчас ~15 мин). Хранить в записи специалиста, не в глобальном конфиге и не в коде.
- Доступность — **повторяющееся недельное расписание**: психолог задаёт интервалы по дням недели (напр. вт 10:00–14:00), редактируемые (добавление/удаление). Хранится в YDB. Слоты генерируются внутри этих интервалов с шагом его длительности.

Бронь (запись клиента):
- Самостоятельная персистентная сущность в YDB с **неизменным UUID** — не «слот в статусе занят». Слоты эфемерны (генерируются из расписания), бронь — нет.
- Хранить **явные `start` и `end` в UTC** на самой брони, а не start + подразумеваемую длительность. При бронировании денормализовать `end`, чтобы позднее изменение длительности психолога не «сдвигало» существующие записи.
- Отмена — **переход состояния (soft), никогда не hard-delete**. Строка брони переживает отмену.

Инварианты:
- Слот существует только внутри окна доступности психолога.
- Один слот нельзя забронировать дважды — проверка при записи обязательна (учитывать гонки).
- Изменение расписания или длительности не должно затирать уже забронированные слоты — их судьбу решать явно (запретить или требовать отмены).
- **Время везде хранить в UTC**, показывать пользователю в его/локальном часовом поясе. Расписание психолога задаётся в его таймзоне — фиксировать её в записи специалиста.
- Состояния слота: свободен / забронирован / отменён.

## Архитектура (serverless-ограничения)
- **Только webhook**, никакого long polling — функция эфемерна.
- Хендлер идемпотентен, не держит фонового состояния между вызовами.
- **Уведомления** — отдельная функция по timer-триггеру YC. Никаких APScheduler / фоновых asyncio-задач: они не переживут завершение вызова.
- Клиент YDB инициализируется на уровне модуля и переиспользуется тёплым инстансом; вручную закрывать не нужно.
- Секреты (токен бота, креды YDB) — только из переменных окружения функции, в код не хардкодить.
- Все переходы состояния брони (создание/отмена/изменение) — через **единый доменный сервисный слой**, не инлайн в aiogram-хендлерах. Хендлеры тонкие; доменная логика тестируется без Telegram.

## Рабочий процесс
Spec-driven (OpenSpec): сначала спека изменения — потом код. Не отступать от активной спеки без её обновления.

## Требования к коду (обязательно)
- **Test-first**: сначала падающий тест, затем реализация. Никакой логики без предшествующего теста. Тесты расположены в директории tests.
- Покрытие ≥ 80%.
- Перед завершением любой задачи обе команды должны быть зелёными:
  - `make test` — тесты
  - `make validate` — линтер + форматтер + тайп-чекер
- Всё исполнять через `uv run` (тесты, скрипты, инструменты).
- Типизировать публичные функции; проходить тайп-чекер без `# type: ignore`-заглушек.

## Не делать
- Не использовать polling.
- Не добавлять планировщики/фоновые задачи внутри функции.
- Не писать код без предшествующего теста.
- Не коммитить креды и токены.
