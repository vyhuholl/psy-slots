## Requirements
### Requirement: Генерация слотов внутри окон доступности с шагом длительности

Система SHALL порождать слоты только внутри интервалов недельного расписания специалиста, начиная от начала интервала с шагом, равным длительности слота специалиста. Слот MUST полностью помещаться в интервал: слот, чей конец выходит за конец интервала, MUST NOT порождаться. Слоты эфемерны и MUST NOT сохраняться в хранилище.

#### Scenario: Слоты идут с шагом длительности

- **WHEN** интервал 10:00–11:00 и длительность 15 минут
- **THEN** порождаются слоты 10:00, 10:15, 10:30, 10:45 (каждый длиной 15 минут)

#### Scenario: Хвостовой слот не порождается

- **WHEN** интервал 10:00–10:50 и длительность 15 минут
- **THEN** порождаются 10:00, 10:15, 10:30, а слот 10:45–11:00 не порождается (выходит за конец интервала)

#### Scenario: Ничего вне окон

- **WHEN** момент времени не попадает ни в один интервал доступности специалиста
- **THEN** для него слот не порождается

### Requirement: Материализация локального времени в UTC

Слоты SHALL материализоваться из локального настенного времени интервалов (в таймзоне специалиста) в конкретные моменты UTC для дат, чей день недели совпадает с днём интервала. Выдаваемые `start` и `end` MUST быть в UTC.

#### Scenario: Локальный слот переводится в UTC

- **WHEN** для конкретной даты порождается слот из интервала в таймзоне специалиста
- **THEN** его `start` и `end` возвращаются в UTC, соответствующих локальному времени этой даты

#### Scenario: Совпадение дня недели

- **WHEN** интервал задан на определённый день недели
- **THEN** слоты порождаются только для дат этого дня недели в запрошенном окне

### Requirement: Политика DST при переходах часов

При переводе локального времени слота в UTC система SHALL применять детерминированную политику переходов DST. Несуществующее локальное время (весенний переход) MUST приводить к пропуску слота. Неоднозначное локальное время (осенний переход) MUST давать ровно один слот (первое вхождение, `fold=0`), без дублирования.

#### Scenario: Несуществующее локальное время пропускается

- **WHEN** локальное начало слота попадает в «дыру» весеннего перехода (такого времени в этот день не существует)
- **THEN** этот слот не порождается

#### Scenario: Неоднозначное локальное время даёт один слот

- **WHEN** локальное начало слота попадает в повторяющийся час осеннего перехода
- **THEN** порождается ровно один слот (первое вхождение), без дубликата

### Requirement: Исключение занятых слотов

Выдача SHALL содержать только свободные слоты. Слот, чей `[start, end)` пересекается с активной (`booked`) бронью того же специалиста, MUST исключаться. Слот, соответствующая бронь которого отменена, MUST снова считаться свободным.

#### Scenario: Занятый слот исключён

- **WHEN** для специалиста есть активная бронь, пересекающая порождённый слот
- **THEN** этот слот не попадает в выдачу свободных слотов

#### Scenario: Отменённая бронь освобождает слот

- **WHEN** бронь на слот отменена
- **THEN** соответствующий слот снова присутствует в выдаче свободных слотов

### Requirement: Запрос свободных слотов по окну

Система SHALL возвращать свободные слоты специалиста, чьё начало попадает в запрошенное окно `[from, to)` в UTC, упорядоченные по возрастанию `start`. При отсутствии доступности или дат в окне выдача MUST быть пустой.

#### Scenario: Фильтрация по окну

- **WHEN** запрошены свободные слоты специалиста в окне `[from, to)`
- **THEN** возвращаются только слоты с `from ≤ start < to`, упорядоченные по `start`

#### Scenario: Пустая выдача

- **WHEN** у специалиста нет интервалов доступности или ни одна их дата не попадает в окно
- **THEN** выдача пуста, без ошибки
