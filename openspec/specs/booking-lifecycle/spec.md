## Requirements
### Requirement: Бронь как персистентная сущность с явным временем в UTC

Система SHALL хранить каждую бронь отдельной строкой в YDB с неизменным идентификатором UUID, идентификатором клиента, а также **явными `start` и `end` в UTC**. `end` MUST денормализоваться при создании как `start + длительность из конфигурации на момент создания` и MUST NOT пересчитываться позднее. Идентификатор MUST NOT изменяться после создания. Время MUST храниться в UTC. Ссылки на специалиста MUST NOT быть — психолог один.

#### Scenario: Создание фиксирует явные start и end

- **WHEN** сервис создаёт бронь на слот с началом `start`
- **THEN** строка сохраняется с сгенерированным UUID, значением `start` в UTC и `end = start + длительность из конфигурации`, обоими в UTC

#### Scenario: Изменение длительности не сдвигает существующие брони

- **WHEN** длительность в конфигурации меняется после создания брони
- **THEN** `end` ранее созданной брони остаётся прежним (значение денормализовано на самой брони)

### Requirement: Бронирование только на сегодня, внутри окна доступности и по сетке слотов

Создание брони SHALL допускаться только если слот приходится на **сегодня** (в таймзоне конфигурации), лежит внутри интервала доступности из конфигурации и выровнен по сетке слотов. Сервис MUST конвертировать `start` из UTC в таймзону конфигурации, проверить, что дата — сегодняшняя и время ещё не наступило, найти интервал доступности, содержащий `[start, end)`, и проверить, что смещение начала от начала интервала кратно длительности. Иначе создание MUST отклоняться.

#### Scenario: Слот сегодня, внутри окна и на сетке — бронируется

- **WHEN** локальное время слота — сегодня, попадает в интервал доступности, выровнено по шагу длительности, `end` не выходит за конец интервала, и время ещё не наступило
- **THEN** бронь создаётся

#### Scenario: Слот не на сегодня — отклоняется

- **WHEN** локальная дата слота не является сегодняшней (другой день или уже прошедшее сегодня время)
- **THEN** создание завершается доменной ошибкой «только на сегодня», бронь не создаётся

#### Scenario: Слот вне окна доступности — отклоняется

- **WHEN** локальное время слота не попадает ни в один интервал доступности
- **THEN** создание завершается доменной ошибкой «вне доступности», бронь не создаётся

#### Scenario: Слот не выровнен по сетке или выходит за интервал — отклоняется

- **WHEN** начало слота не кратно длительности от начала интервала либо `end` выходит за конец интервала
- **THEN** создание завершается доменной ошибкой, бронь не создаётся

### Requirement: Запрет двойного бронирования с учётом гонок

Система SHALL запрещать создание брони, чьё `[start, end)` пересекается с любой **активной** (`booked`) бронью. Проверка и вставка MUST выполняться атомарно в сериализуемой транзакции, так что при конкурентных попытках забронировать один слот успешна не более чем одна. Отменённые брони MUST NOT препятствовать повторному бронированию того же слота.

#### Scenario: Пересечение с активной бронью

- **WHEN** уже есть активная бронь, пересекающаяся с запрашиваемым `[start, end)`
- **THEN** создание завершается доменной ошибкой «слот занят», вторая бронь не создаётся

#### Scenario: Конкурентное создание одного слота

- **WHEN** две попытки создать бронь на один и тот же слот выполняются одновременно
- **THEN** ровно одна завершается успехом, другая — доменной ошибкой «слот занят»

#### Scenario: Повторное бронирование после отмены

- **WHEN** предыдущая бронь на слот отменена, и слот запрашивается снова
- **THEN** новая бронь создаётся успешно (отменённая строка не блокирует слот)

### Requirement: Отмена как soft-переход состояния

Отмена SHALL быть переходом состояния `booked → cancelled` и MUST NOT удалять строку (никакого hard-delete): строка брони переживает отмену и остаётся доступной для чтения. Отмена MUST быть идемпотентной — повторная отмена уже отменённой брони не производит второго эффекта и не является ошибкой.

#### Scenario: Отмена активной брони

- **WHEN** сервис отменяет бронь в состоянии `booked`
- **THEN** её состояние становится `cancelled`, строка сохраняется и остаётся читаемой по id

#### Scenario: Идемпотентная повторная отмена

- **WHEN** сервис отменяет бронь, уже находящуюся в состоянии `cancelled`
- **THEN** операция завершается успешно без ошибки, состояние остаётся `cancelled`, второго наблюдаемого эффекта нет

#### Scenario: Отмена несуществующей брони

- **WHEN** сервис пытается отменить бронь по неизвестному id
- **THEN** операция завершается доменной ошибкой «не найдено»

### Requirement: Единый доменный сервис и чтение броней

Все переходы состояния брони (создание, отмена) SHALL проходить через доменный сервис, тестируемый без Telegram; переходы MUST NOT выполняться инлайн в aiogram-хендлерах. Состояния брони — `booked` и `cancelled`; слот считается свободным при отсутствии активной брони. Сервис SHALL предоставлять чтение: получение по id, список активных броней в диапазоне времени, список активных броней клиента.

#### Scenario: Получение по id

- **WHEN** сервис запрашивает бронь по существующему id
- **THEN** возвращается бронь с её текущим состоянием; по неизвестному id — явный результат «не найдено»

#### Scenario: Список активных исключает отменённые

- **WHEN** сервис запрашивает активные брони в диапазоне времени
- **THEN** возвращаются только брони в состоянии `booked`, попадающие в диапазон, и не возвращаются отменённые

#### Scenario: Список активных броней клиента

- **WHEN** сервис запрашивает активные брони клиента
- **THEN** возвращаются только его брони в состоянии `booked`
