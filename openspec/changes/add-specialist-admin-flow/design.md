## Context

Сервисы специалиста, расписания и броней готовы, клиентский поток дал общие `app/bot/` утилиты (форматирование, клавиатуры, `callback_data`, тонкие хендлеры без серверного состояния). Не хватает управляющего интерфейса психолога и единственного места, где живёт инвариант «изменение расписания/длительности не должно затирать забронированные слоты — их судьбу решать явно». Предыдущие changes сознательно отложили этот гвард сюда, где сходятся правка параметров и брони.

Для авторизации нужно понимать, какой Telegram-пользователь — какой специалист, поэтому запись специалиста расширяется привязкой `telegram_id`.

## Goals / Non-Goals

**Goals:**

- Привязка специалиста к аккаунту Telegram (`telegram_id`, уникальный) и поиск по нему.
- Авторизация: админ-действия только для привязанного специалиста.
- Тонкие хендлеры поверх сервисов: длительность, расписание (add/remove), просмотр и отмена броней.
- Гвард инвариантности: блокировать изменение длительности и удаление интервала при наличии затрагиваемых активных будущих броней (требовать отмены).
- Уведомление клиента при отмене брони специалистом.
- Время — в таймзоне специалиста. Тестируемость без реального Telegram.

**Non-Goals:**

- Никакой доменной логики броней/расписания заново — оркестрация существующих сервисов.
- Никакого self-service «регистрации» нового специалиста через бота — наполнение реестра остаётся операцией сервиса/сида.
- Никакого планировщика напоминаний (см. `add-reminder-notifications`) — здесь только разовое уведомление клиента как прямое сообщение при отмене.
- Никакого управления чужими записями — только своя запись специалиста.

## Decisions

**Привязка `telegram_id` на записи специалиста — дельта `specialist-registry`.** Добавляем колонку `telegram_id Int64?` (nullable, уникальный) в `specialists` (идемпотентный `ALTER TABLE ADD COLUMN` + вторичный индекс) и метод `get_by_telegram_id`. Привязка выполняется операционно (сид/админ задаёт `telegram_id` записи) — это данные, а не хардкод. Self-service привязка по коду возможна позже. Альтернатива (список admin-id в env) отвергнута: смешивает конфиг с доменной идентичностью и хуже масштабируется на переменное число специалистов.

**Авторизация в хендлере — по `get_by_telegram_id`.** Каждый админ-хендлер сначала резолвит специалиста по `telegram_id` отправителя; нет привязки → отказ, действий не показываем. Единая проверка на входе роутера (middleware/фильтр), чтобы не дублировать.

**Гвард — композит поверх read-запросов броней, политика «требовать отмены» (блокировка).** Инвариант допускает «запретить или требовать отмены»; выбираем блокировку с требованием предварительной отмены — самое явное и безопасное разрешение.

- *Длительность*: изменение затрагивает всю будущую сетку слотов. Перед применением запрашиваем `booking_service.list_active_for_specialist(from=now, to=∞-горизонт)`; если непусто — блок с перечнем конфликтных броней. Существующие брони не трогаем (их `end` денормализован — они не сдвигаются), но меняем длительность только когда будущих активных броней нет.
- *Удаление интервала*: запрашиваем активные будущие брони специалиста и фильтруем те, чьё локальное время попадает в этот интервал по дню недели `[start,end)`; если есть — блок. Иначе удаляем.
- *Добавление интервала*: безопасно (только расширяет доступность), гвард не нужен; пересечения уже ловит `availability_service`.

Логику гварда держим в тонком доменном хелпере (`app/services/schedule_guard.py` или методах существующих сервисов), не инлайн в хендлере, чтобы тестировать без Telegram. Альтернатива (тихо переприменять/сдвигать брони) отвергнута — нарушает инвариант.

**Отмена брони специалистом + уведомление клиента.** `booking_service.cancel` (идемпотентный soft-переход), затем прямое `bot.send_message(client_id, …)` об отмене (у нас есть `client_id`=chat_id и TZ клиента из `client-profile` для форматирования). Это разовое уведомление, не планировщик — согласуется с serverless-ограничениями. Полноценные напоминания — отдельный change.

**Навигация и хендлеры — как в клиентском потоке.** Состояние в `callback_data` (`adm:duration`, `adm:sched`, `adm:addint:…`, `adm:delint:<interval_id>`, `adm:bookings`, `adm:cancel:<booking_id>` и т.п.), тонкие хендлеры в `app/bot/handlers/admin.py`, переиспользование `formatting.py`/`keyboards.py`. Время — в TZ специалиста (из его записи). Тесты — прямые вызовы хендлеров с поддельными `Message`/`CallbackQuery` и замоканными сервисами.

## Risks / Trade-offs

- **Строгая политика гварда** (блок при любых будущих бронях) может раздражать при рутинной правке. Mitigation: чёткое сообщение с перечнем и путём (отменить брони); при необходимости позже смягчить до «предупредить и подтвердить» — точка расширения зафиксирована.
- **Горизонт «будущих» броней** для длительности формально бесконечен. Mitigation: практический горизонт (напр. как у показа слотов) + запрос по индексу `(specialist_id, start_utc)`; активных будущих броней немного.
- **Сопоставление рекуррентного интервала и конкретных броней** для удаления. Mitigation: берём активные будущие брони специалиста, конвертируем `start_utc`→TZ специалиста, сравниваем `weekday` и минуту суток с интервалом; объёмы малы.
- **Уведомление клиента может не дойти** (клиент заблокировал бота). Mitigation: отмена всё равно фиксируется (источник истины — бронь); ошибку отправки логируем, но не откатываем отмену.
- **Bootstrap привязки** (первый `telegram_id`) — операционный шаг. Mitigation: задаётся сидом/сервисом; отмечено в Open Questions.

## Migration Plan

1. TDD-порядок: дельта реестра (падающие тесты `telegram_id`, уникальность, `get_by_telegram_id`) → расширение домена/сервиса + `ALTER TABLE`; затем гвард (падающие тесты блокировок) → `schedule_guard`; затем авторизация и хендлеры (падающие тесты с моками) → `handlers/admin.py`; подключение роутеров.
2. Прогнать миграции — добавляется колонка `telegram_id` и индекс.
3. Привязать существующих специалистов (сид: проставить `telegram_id`).
4. Smoke: привязанный специалист меняет длительность (успех/блок), правит расписание, отменяет бронь → клиент получает уведомление.

**Rollback:** снять регистрацию админ-роутеров (управление недоступно, данные целы); колонка `telegram_id` nullable и идемпотентна, переживает редеплой.

## Open Questions

- Механизм первичной привязки `telegram_id` (сид vs self-service по одноразовому коду) — по умолчанию сид; self-service при потребности.
- Смягчать ли гвард до «предупредить и подтвердить» вместо жёсткой блокировки — по умолчанию блокировка; решается по обратной связи.
- Текст и объём уведомления клиенту при отмене (с предложением перезаписаться?) — согласовать с UX.
