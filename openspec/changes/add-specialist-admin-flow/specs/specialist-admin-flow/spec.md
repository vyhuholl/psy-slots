## ADDED Requirements

### Requirement: Авторизация специалиста

Команды админ-потока SHALL быть доступны только пользователю Telegram, привязанному к записи специалиста. Пользователь без привязки MUST получать отказ и MUST NOT видеть управляющих действий.

#### Scenario: Привязанный специалист получает доступ

- **WHEN** пользователь с `telegram_id`, привязанным к специалисту, открывает админ-меню
- **THEN** ему показываются управляющие действия для его записи

#### Scenario: Непривязанный пользователь получает отказ

- **WHEN** пользователь без привязки к специалисту вызывает админ-команду
- **THEN** бот отвечает отказом, управляющие действия не показываются

### Requirement: Изменение длительности слота с гвардом по активным броням

Специалист SHALL изменять длительность своего слота через доменный сервис. Если у специалиста есть активные будущие брони, изменение длительности MUST быть заблокировано с указанием отменить их предварительно; система MUST NOT затирать или сдвигать существующие брони.

#### Scenario: Изменение длительности без активных броней

- **WHEN** привязанный специалист меняет длительность и активных будущих броней нет
- **THEN** новая длительность сохраняется через сервис

#### Scenario: Блокировка при активных бронях

- **WHEN** специалист пытается изменить длительность при наличии активных будущих броней
- **THEN** изменение отклоняется с сообщением отменить эти брони, длительность не меняется, существующие брони сохраняются неизменными

### Requirement: Редактирование недельного расписания

Специалист SHALL добавлять и удалять интервалы своего расписания через доменный сервис. Добавление корректного интервала MUST быть разрешено. Удаление интервала, внутри будущих вхождений которого есть активные брони, MUST быть заблокировано с указанием отменить их предварительно.

#### Scenario: Добавление интервала

- **WHEN** специалист добавляет корректный непересекающийся интервал
- **THEN** интервал сохраняется через сервис и появляется в расписании

#### Scenario: Удаление свободного интервала

- **WHEN** специалист удаляет интервал, внутри будущих вхождений которого нет активных броней
- **THEN** интервал удаляется

#### Scenario: Блокировка удаления интервала с бронями

- **WHEN** специалист пытается удалить интервал, внутри будущих вхождений которого есть активные брони
- **THEN** удаление отклоняется с указанием отменить эти брони, интервал сохраняется

### Requirement: Просмотр броней специалиста

Специалист SHALL видеть список своих активных броней со временем, сконвертированным в его таймзону.

#### Scenario: Список броней

- **WHEN** специалист запрашивает свои брони
- **THEN** бот показывает активные брони со временем в таймзоне специалиста

#### Scenario: Пустой список

- **WHEN** у специалиста нет активных броней
- **THEN** бот сообщает об отсутствии записей без ошибки

### Requirement: Отмена брони специалистом

Специалист SHALL отменять бронь своей записи через доменный сервис как soft-переход (без hard-delete). При отмене клиент MUST быть уведомлён.

#### Scenario: Отмена и уведомление клиента

- **WHEN** специалист отменяет активную бронь
- **THEN** бронь переводится в состояние «отменена» через сервис, и клиенту отправляется уведомление об отмене

#### Scenario: Отмена уже отменённой брони

- **WHEN** специалист отменяет бронь, уже находящуюся в состоянии «отменена»
- **THEN** операция завершается без ошибки и без повторного эффекта (идемпотентно)
