## Why

Клиенту нужно показать конкретные свободные слоты, но слоты не хранятся — они эфемерны и выводятся из недельного расписания с шагом длительности специалиста. Это связующая read-модель между расписанием, длительностью и бронями: она материализует абстрактные окна доступности в конкретные слоты в UTC и вычитает уже занятые. Без неё клиентский поток записи не из чего строить.

## What Changes

- Чистая генерация слотов: внутри каждого интервала доступности — слоты с шагом = длительность специалиста, без «хвостового» слота, выходящего за конец интервала.
- **Материализация локального времени в UTC**: интервалы заданы в таймзоне специалиста; для конкретных дат диапазона слоты переводятся в UTC.
- **Политика DST** (единственное место, где решается локальное→UTC): несуществующее локальное время (весенний переход) — слот пропускается; неоднозначное (осенний переход) — один слот (`fold=0`), без дубля.
- **Исключение занятых**: слоты, пересекающиеся с активной бронью, не попадают в выдачу; слот отменённой брони снова свободен.
- Запрос по окну `[from, to)` в UTC; выдача упорядочена по началу; слоты нигде не персистятся.
- Разделение на чистое ядро (математика слотов + DST, без I/O) и сервис (подтягивает специалиста, расписание, брони). Тестируется без Telegram.

## Impact

- **Новый код**: `app/domain/slots.py` (чистое ядро: `Slot`, генерация из интервалов+длительности+TZ+окна, DST), `app/services/slot_service.py` (оркестрация над реестром, расписанием, бронями).
- **Тесты**: `tests/` — шаг и границы слота (без хвостового), материализация в UTC, DST (несуществующее/неоднозначное локальное время), исключение занятых, фильтр по окну, пустая выдача.
- **Зависит от**: `add-specialist-registry` (длительность + TZ), `add-weekly-availability` (интервалы), `add-booking-lifecycle` (активные брони для исключения).
- **Разблокирует**: `add-client-booking-flow` (показ и выбор свободных слотов).
- **Без персистентности**: миграций нет; слоты вычисляются на лету.
- **Консистентность**: выдача — совещательная; бронь могла появиться между генерацией и записью, поэтому `add-booking-lifecycle` перепроверяет окно и защищает от двойного бронирования при записи.
