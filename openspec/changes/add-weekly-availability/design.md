## Context

`add-specialist-registry` дал сущность психолога с его таймзоной и доменный сервис. Следующий слой — доступность: повторяющееся недельное расписание, внутри окон которого позже генерируются слоты (инвариант: слот существует только внутри окна доступности). По CLAUDE.md расписание задаётся в таймзоне специалиста и редактируется им (добавление/удаление интервалов).

Ключевой момент — как хранить время. Расписание повторяется еженедельно в именованной зоне, поэтому хранить его в UTC нельзя: смещение зоны меняется на переходах DST, и «вторник 10:00» должен оставаться 10:00 по местному в любую неделю. Значит, храним локальное настенное время + день недели, а перевод в конкретные UTC-моменты — забота генерации слотов.

Здесь только домен + персистентность расписания. Telegram-UI редактирования — `add-specialist-admin-flow`.

## Goals / Non-Goals

**Goals:**

- Таблица YDB интервалов расписания через существующий раннер миграций.
- Неизменяемая типизированная модель интервала.
- Доменный сервис как единственный вход: add / remove / list.
- Валидация: день недели, начало < конец, границы суток, существование специалиста, запрет пересечений в один день.
- Тестируемость без Telegram и без сети.

**Non-Goals:**

- Никаких aiogram-хендлеров и UX (см. `add-specialist-admin-flow`).
- Никакой генерации слотов и перевода в UTC (см. `add-slot-generation`).
- Никакого разрешения конфликта «удаление/правка интервала vs существующие брони» — броней ещё нет; guard вводится в `add-booking-lifecycle`, где доступны и брони, и это расписание.
- Никакой поддержки интервалов через полночь (конец < начало) — вне области; при необходимости разбивается на два дня позже.

## Decisions

**Хранение времени — минуты от полуночи + день недели, в TZ специалиста.** Колонки `weekday Uint8`, `start_minute Uint16`, `end_minute Uint16` (0..1440). Минуты от полуночи — просто сравнивать, легко считать шаг слота, нет неоднозначности парсинга. Настенное локальное время, НЕ UTC — по причине DST выше. Альтернатива (хранить `Timestamp`/UTC) ломает еженедельную повторяемость и переходы DST.

**День недели — Monday=0 … Sunday=6** (как `datetime.weekday()`). Совпадает со стандартной библиотекой Python, упрощает будущую генерацию слотов (сопоставление даты и дня недели без конвертаций). Валидный диапазон 0..6.

**Схема `availability_intervals`.** `id Utf8` (PK, UUID-строка), `specialist_id Utf8`, `weekday Uint8`, `start_minute Uint16`, `end_minute Uint16`. Отдельный `id` нужен, т.к. у специалиста несколько интервалов в один день — удаление по идентификатору. Чтение по `specialist_id` (вторичный индекс или скан по фильтру — при малых объёмах приемлемо; индекс добавить при росте).

**Границы модулей.** `app/domain/availability.py` — frozen `@dataclass AvailabilityInterval` (specialist_id, weekday, start_minute, end_minute) + доменные ошибки (`ValidationError`, `IntervalOverlap`) и переиспользование `SpecialistNotFound` из домена специалиста; чистая валидация формы интервала в модели. `app/services/availability_service.py` — сервис над YDB: проверяет существование специалиста (через `SpecialistService`/лукап), проверяет пересечения, генерирует UUID, маппит строку⇄модель. Модель без YDB, сервис без Telegram.

**Проверка пересечений в сервисе, не в БД.** Перед вставкой сервис читает интервалы специалиста на этот день недели и проверяет перекрытие полуоткрытых диапазонов `[start, end)`: пересечение ⟺ `a.start < b.end and b.start < a.end`. Смежные (`end == start`) разрешены. YDB не выражает такой constraint декларативно; проверка в сервисе достаточна при редком редактировании расписания. Гонки при одновременной правке одним специалистом маловероятны (last-write-wins приемлем для конфига) — отмечено в рисках.

**Существование специалиста — обязательное условие интервала.** Сервис доступности зависит от сервиса специалиста и отклоняет добавление для несуществующего id (`SpecialistNotFound`). Так поддерживается инвариант «интервал принадлежит реальному специалисту».

**Удаление интервала — реальное удаление строки.** Интервалы расписания — редактируемый конфиг, а не персистентная доменная сущность: требование «никогда не hard-delete» относится только к броням. Поэтому remove — обычное удаление. Guard «нельзя удалить интервал с активными бронями внутри» появится в `add-booking-lifecycle`, где брони уже существуют и их можно запросить.

## Risks / Trade-offs

- **Интервалы через полночь не поддержаны** (требуют `end < start`). Mitigation: явный non-goal; при появлении ночных приёмов — разбивать на два дня в UI/сервисе.
- **Пересечения проверяются в приложении, есть окно гонки** при параллельной правке одной пары специалист+день. Mitigation: правит один специалист, частота низкая; при необходимости — транзакция/сериализация в YDB. last-write-wins приемлем для конфига.
- **DST-«дыры»/«дубли»** (перевод часов): 10:00 в день перевода может не существовать или существовать дважды. Mitigation: здесь только хранение локального времени; корректная материализация — ответственность `add-slot-generation` (там политика для несуществующего/неоднозначного локального времени).
- **Скан по `specialist_id` без индекса** на больших объёмах. Mitigation: специалистов и интервалов немного; добавить вторичный индекс при росте.
- **Конфликт с бронями при правке** не проверяется здесь. Mitigation: явный non-goal + зависимость; инвариант закрывается в `add-booking-lifecycle`.

## Migration Plan

1. TDD-порядок: сначала падающие тесты модели (день недели, границы, начало<конец), затем модель; затем тесты сервиса (add/remove/list, пересечения, несуществующий специалист) против замоканного пула, затем сервис; затем регистрация таблицы в раннере миграций.
2. Прогнать обновлённые миграции против serverless-YDB отдельным шагом — создаётся `availability_intervals`.

**Rollback:** таблица не используется другими change до их реализации; `IF NOT EXISTS` идемпотентен и переживает редеплой; откат кода не требует разрушения таблицы.
