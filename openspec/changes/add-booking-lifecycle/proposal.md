## Why

Бронь — центральная сущность продукта и единственная, чьё состояние переживает всё остальное. По CLAUDE.md это самостоятельная персистентная запись с неизменным UUID, а не «слот в статусе занят»: слоты эфемерны, бронь — нет. На ней держатся ключевые инварианты (слот только внутри окна доступности, запрет двойного бронирования с учётом гонок, отмена только soft). Пока нет сущности брони и единого доменного сервиса её переходов, ни запись клиента, ни генерация свободных слотов, ни напоминания невозможны.

## What Changes

- Новая таблица YDB `bookings`: неизменный UUID, специалист, клиент, **явные `start` и `end` в UTC**, состояние, метки времени.
- Доменная модель брони (типизированная, неизменяемая) и **единый доменный сервис** всех переходов: создание и отмена. Никаких переходов инлайн в aiogram-хендлерах.
- **Денормализация `end`** при создании: `end = start + длительность специалиста` фиксируется на брони, чтобы позднее изменение длительности не «сдвигало» существующие записи.
- **Проверка окна доступности** при создании: `start` конвертируется из UTC в TZ специалиста и должен попадать в интервал недельного расписания и на сетку слотов (шаг = длительность).
- **Запрет двойного бронирования с учётом гонок**: пересекающаяся активная бронь отклоняется; конкурентное создание одного слота проходит не более чем в одном экземпляре (сериализуемая транзакция YDB).
- **Отмена — soft-переход** `booked → cancelled`, идемпотентный, никогда не hard-delete: строка брони переживает отмену.
- Состояния: свободен (нет активной брони) / забронирован / отменён. «Изменение» брони реализуется как отмена + новая бронь, без правки времени на месте.
- Read-операции сущности: получение по id, список активных броней специалиста в диапазоне, список активных броней клиента.
- Сервис тестируется без Telegram; никаких aiogram-хендлеров в этом change.

## Capabilities

## Impact

- **Новый код**: `app/domain/booking.py` (модель, состояния, ошибки), `app/services/booking_service.py` (создание/отмена/чтение над YDB в транзакции), регистрация таблицы `bookings` в `app/migrations.py`.
- **Тесты**: `tests/` — денормализация end, валидация окна/сетки, отклонение вне доступности, отклонение пересечений, идемпотентная отмена, переживание строки после отмены, чтение.
- **Зависит от**: `add-specialist-registry` (длительность + TZ), `add-weekly-availability` (окна расписания), `bootstrap-bot` (YDB, миграции).
- **Разблокирует**: `add-slot-generation` (исключает забронированное), `add-client-booking-flow` (запись/отмена клиентом), `add-specialist-admin-flow` (гвард «правка длительности/расписания vs активные брони» — через read-запросы отсюда), `add-reminder-notifications` (сканирует ближайшие брони).
- **DST**: сторона брони чиста — `start` хранится в UTC, а перевод UTC→локальное для проверки окна однозначен; неоднозначность локального→UTC остаётся заботой `add-slot-generation`.
