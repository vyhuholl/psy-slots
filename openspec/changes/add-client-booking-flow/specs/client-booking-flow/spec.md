## ADDED Requirements

### Requirement: Пошаговый выбор без серверного состояния

Клиент SHALL записываться через последовательный выбор специалиста, затем дня, затем свободного слота, реализованный инлайн-клавиатурами. Состояние навигации MUST кодироваться в `callback_data` кнопок; система MUST NOT полагаться на серверное/фоновое состояние между вызовами функции.

#### Scenario: Выбор специалиста показывает дни

- **WHEN** клиент выбирает специалиста из списка
- **THEN** бот показывает дни с доступностью для этого специалиста

#### Scenario: Выбор дня показывает свободные слоты

- **WHEN** клиент выбирает день
- **THEN** бот показывает свободные слоты этого дня, полученные из сервиса генерации слотов

#### Scenario: Состояние берётся из callback_data

- **WHEN** клиент нажимает кнопку слота
- **THEN** выбор специалиста и времени восстанавливается из `callback_data` без обращения к серверной сессии

### Requirement: Подтверждение создаёт бронь через доменный сервис

Подтверждение выбранного слота SHALL создавать бронь через доменный сервис броней. Хендлеры MUST быть тонкими: доменная логика MUST NOT выполняться инлайн в хендлере.

#### Scenario: Успешное подтверждение

- **WHEN** клиент подтверждает свободный слот
- **THEN** через доменный сервис создаётся бронь, и клиент получает подтверждение записи

#### Scenario: Повторное подтверждение не создаёт вторую бронь

- **WHEN** подтверждение одного и того же слота приходит повторно (двойное нажатие или повторная доставка апдейта)
- **THEN** вторая бронь не создаётся, и клиент видит согласованный результат (его существующую бронь)

### Requirement: Отображение времени в таймзоне клиента

Все показываемые клиенту время и даты (дни, слоты, подтверждения, список броней) SHALL быть сконвертированы из UTC в таймзону клиента из его профиля.

#### Scenario: Слоты во времени клиента

- **WHEN** бот показывает свободные слоты или подтверждение брони
- **THEN** время отображается в таймзоне клиента, а не в UTC и не в таймзоне специалиста

### Requirement: Обработка занятого слота

Если к моменту подтверждения слот оказался занят (гонка с другой бронью), система SHALL сообщить об этом клиенту без сбоя и MUST NOT создавать вторую бронь.

#### Scenario: Слот занят к моменту подтверждения

- **WHEN** клиент подтверждает слот, который уже был забронирован между показом и подтверждением
- **THEN** бот показывает понятное сообщение «слот занят» и предлагает выбрать другой, без исключения и без второй брони

### Requirement: Просмотр и отмена своих броней

Клиент SHALL видеть список своих активных броней и отменять любую из них. Отмена MUST выполняться через доменный сервис как soft-переход (без hard-delete).

#### Scenario: Список своих броней

- **WHEN** клиент запрашивает свои записи
- **THEN** бот показывает его активные брони со временем в таймзоне клиента

#### Scenario: Отмена брони

- **WHEN** клиент отменяет свою бронь
- **THEN** через доменный сервис бронь переводится в состояние «отменена», и клиент получает подтверждение отмены

#### Scenario: Пустой список

- **WHEN** у клиента нет активных броней
- **THEN** бот сообщает об отсутствии записей без ошибки
