## Context

Доменный слой полностью готов: `specialist_service` (список специалистов, длительность, TZ), `slot_service` (свободные слоты в UTC), `booking_service` (создание/отмена/чтение, race-safe). `bootstrap-bot` дал Dispatcher в webhook-режиме и идемпотентный, stateless-хендлер. Нужен первый пользовательский поток: клиент записывается и управляет своими бронями, видя время в своём часовом поясе.

Два ограничения формируют дизайн: (1) функция эфемерна и не держит состояние между вызовами — значит, никакого серверного FSM; (2) CLAUDE.md требует показывать время в TZ клиента — значит, нужен минимальный профиль клиента с таймзоной.

## Goals / Non-Goals

**Goals:**

- Тонкие aiogram-хендлеры, вся доменная логика — через существующие сервисы.
- Пошаговый выбор специалиста → день → слот → подтверждение, полностью на инлайн-клавиатурах.
- Навигация без серверного состояния: контекст в `callback_data`.
- Минимальный профиль клиента с таймзоной; запрос при первом контакте; показ времени в этой TZ.
- Создание брони при подтверждении; дружелюбная обработка занятого слота (гонка) и повторного подтверждения.
- Просмотр и soft-отмена своих броней.
- Тестируемость хендлеров без реального Telegram (замоканные сервисы, поддельные `Message`/`CallbackQuery`).

**Non-Goals:**

- Никакой доменной логики броней/слотов заново — только оркестрация существующих сервисов.
- Никакого админ-потока специалиста (см. `add-specialist-admin-flow`).
- Никаких напоминаний (см. `add-reminder-notifications`) — но профиль клиента здесь готовит для них TZ.
- Никакого богатого профиля клиента (имя/контакты/история) — только то, что нужно для показа времени.
- Никакого перевода/локализации помимо форматирования времени.

## Decisions

**Навигация без FSM — состояние в `callback_data`.** Каждая инлайн-кнопка несёт полный контекст шага: `spec:<specialist_id>`, `day:<specialist_id>:<yyyy-mm-dd>`, `slot:<specialist_id>:<start_epoch>`, `confirm:<specialist_id>:<start_epoch>`, `cancel:<booking_id>`, `tz:<name>`. Хендлер восстанавливает выбор из `callback_data` и заново зовёт сервисы — никакого серверного FSM/сессии, что корректно для эфемерной функции. Бюджет `callback_data` — 64 байта: `confirm:` (8) + UUID (36) + `:` + epoch-seconds (~10) ≈ 55 байт, помещается. Альтернатива (aiogram FSM со внешним storage в YDB) добавляет состояние и запись на каждый шаг — избыточно и противоречит stateless-хендлеру.

**Профиль клиента — таблица `clients`.** `telegram_id Int64` (PK), `timezone Utf8` (IANA), `created_at Timestamp`. `app/domain/client.py` (валидация TZ через `zoneinfo`) + `app/services/client_service.py` (`get`, `set_timezone`). При первом контакте без TZ бот показывает выбор из нескольких распространённых зон (кнопки `tz:<name>`) и сохраняет выбор; далее TZ берётся из профиля. `telegram_id` совпадает с `client_id` брони и chat_id для будущих напоминаний.

**Дефолт таймзоны.** Если клиент пропустил выбор, используем конфигурируемую дефолтную TZ (env), пока он её не задаст; сменить можно командой `/timezone`. Так поток не блокируется, но требование «показывать в TZ клиента» выполняется, как только она известна.

**Границы модулей.** `app/bot/handlers/client.py` — тонкие хендлеры команд (`/book`, `/mybookings`, `/timezone`) и колбэков; `app/bot/keyboards.py` — сборка клавиатур и единая (де)сериализация `callback_data`; `app/bot/formatting.py` — форматирование UTC→TZ клиента. Хендлеры только: парсят апдейт, зовут сервис, строят ответ. Роутеры подключаются к Dispatcher из `bootstrap-bot`.

**Окно слотов для дня.** Выбранный день клиента переводится в UTC-окно `[day_start, day_end)` (в TZ клиента), которое передаётся в `slot_service`. Показ будущего — вызывающий передаёт `from = max(now, day_start)`. Дни для показа — ближайшие N (конфигурируемо), где у специалиста есть слоты.

**Обработка занятого слота и повтора.** Подтверждение зовёт `booking_service.create`; на `SlotTaken` хендлер показывает «слот занят, выберите другой» без исключения. Повторное подтверждение (двойной тап/повтор вебхука): защита от двойного бронирования — на стороне `booking-lifecycle`; хендлер при конфликте показывает уже существующую бронь клиента как результат, не создавая вторую.

**Тестирование хендлеров.** Хендлеры вызываются напрямую с поддельными `Message`/`CallbackQuery` и замоканными сервисами; проверяем корректные вызовы сервисов, содержимое клавиатур и форматирование времени в TZ. Реальный Telegram не нужен — доменная логика уже покрыта в своих changes.

## Risks / Trade-offs

- **Лимит `callback_data` 64 байта.** Mitigation: epoch-seconds + UUID укладывается (~55 байт); единая точка (де)сериализации в `keyboards.py`; при росте полей — короткий код специалиста вместо UUID.
- **Telegram не даёт таймзону пользователя.** Mitigation: явный выбор из списка (+ дефолт из env), команда `/timezone` для смены; сохраняем в профиль.
- **Устаревшие клавиатуры** (клиент жмёт слот из старого сообщения, который уже занят). Mitigation: `callback_data` несёт `start_utc`, `booking-lifecycle` перепроверяет окно и занятость при записи; занятый слот → дружелюбная ошибка.
- **Гонка «двойной тап»** создаёт две попытки записи. Mitigation: идемпотентность вебхука + race-safe guard домена; на один слот проходит одна бронь.
- **DST при показе** (несуществующее/неоднозначное локальное время у клиента). Mitigation: показываем существующие слоты (их даёт `slot_service`), форматирование UTC→TZ клиента однозначно.

## Migration Plan

1. TDD-порядок: сначала профиль клиента (падающие тесты валидации TZ и `set/get`) → домен+сервис+таблица `clients`; затем `callback_data` (де)сериализация и клавиатуры (падающие тесты) → `keyboards.py`; затем форматирование в TZ → `formatting.py`; затем хендлеры (падающие тесты с моками) → `handlers/client.py`; подключение роутеров к Dispatcher.
2. Прогнать миграции — создаётся `clients`.
3. Smoke: `/book` → выбор до подтверждения → бронь; `/mybookings` → отмена.

**Rollback:** снять регистрацию клиентских роутеров (бот перестаёт принимать записи); таблица `clients` идемпотентна и переживает редеплой; данные броней не затрагиваются.
