## Context

Бот записи к одному психологу. Домен броней уже реализован: `BookingService.create` валидирует «сегодня + окно + сетка» и защищает от двойного бронирования сериализуемой транзакцией YDB (`_SELECT_OVERLAP` + `_UPSERT_BOOKING` внутри `retry_tx_sync`). Клиентский поток — тонкие хендлеры (`app/bot/handlers/client.py`), уже уведомляющие администратора об отмене клиентом (шаблон try/except pass, `resolve_client_name`). Напоминания — отдельная YC-функция по timer-триггеру (`app/notify.py` → `ReminderService.send_pending`), шлёт клиенту одно идемпотентное напоминание за фиксированные 5 минут; идемпотентность — маркер в таблице `sent_reminders` (PK `booking_id`, claim-then-send).

Три изменения встраиваются в существующие слои без новых зависимостей, таблиц и переменных окружения (`ADMIN_TELEGRAM_ID` уже есть в `Config`).

## Goals / Non-Goals

**Goals:**
- Гарантировать не более одной активной брони на клиента, устойчиво к гонкам.
- Уведомлять администратора о каждой новой броне и за 5 минут до начала каждой сессии.
- Сохранить идемпотентность клиентского напоминания и «сбой уведомления админу не ломает основную операцию».

**Non-Goals:**
- Не менять схему БД, конфигурацию, набор переменных окружения.
- Не вводить отдельный маркер идемпотентности для админского напоминания.
- Не ограничивать брони по календарным дням иначе, чем через «одна активная бронь» (брони и так только на сегодня).
- Не трогать поток отмены (существующее уведомление админа об отмене остаётся как есть).

## Decisions

### 1. Семантика «один слот на клиента» = не более одной активной (`booked`) брони

«Занять только один слот» трактуется как «иметь не более одной активной брони одновременно». Так как бронирование и так только на сегодня, это эквивалентно «один слот в день». Учитываются только `booked`; `cancelled` не считается — после отмены клиент записывается снова. Бронь, чьё время уже прошло, но не отменена, всё ещё `booked` и считается активной (авто-завершения брони в домене нет — не вводим его в этом изменении).

_Альтернатива:_ считать «активной» только будущую бронь (start > now). Отклонено: сложнее, зависит от `now`, и для today-only бота выигрыша не даёт.

### 2. Проверка ограничения — внутри той же транзакции создания

Проверку «нет активной брони у клиента» добавляем в `_insert(tx)` рядом с проверкой пересечения, отдельным `SELECT ... WHERE status='booked' AND client_id=$client_id`. Обе проверки и вставка — в одной `QuerySerializableReadWrite`-транзакции, поэтому конкурентная вторая попытка того же клиента проиграет коммит (`ydb.issues.Aborted`) — её тоже маппим в доменную ошибку.

_Альтернатива:_ отдельный запрос до транзакции. Отклонено: не атомарно, окно гонки между проверкой и вставкой.

### 3. Новая доменная ошибка `ClientAlreadyBooked(BookingError)`

Отдельный подкласс `BookingError`, отличимый от `SlotTaken`. Порядок проверок в транзакции: сначала пересечение слота (`SlotTaken`), затем активная бронь клиента (`ClientAlreadyBooked`) — чтобы «слот занят другим» имело приоритет над «у тебя уже есть запись». Для конкурентной гонки (`Aborted`) выбираем маппинг: победитель по времени определяет, но проигравшему безопаснее вернуть `SlotTaken`, если он бился за тот же слот, иначе `ClientAlreadyBooked`. Практически проигравший `Aborted` не знает причину; маппим `Aborted` в `SlotTaken` как сейчас (это существующее поведение и корректно для конкуренции за один слот). Гонка «один клиент, два разных слота» редка; её проигравший получит `SlotTaken` — приемлемо (сообщение мягкое, повтор покажет `ClientAlreadyBooked`).

### 4. Уведомление админа о новой броне — в хендлере `handle_confirm`, best-effort

После успешного `create` в `handle_confirm` добавляем блок по образцу существующего уведомления об отмене: `resolve_client_name(bot, client_id)` + `format_slot_range(...)`, `bot.send_message(config.admin_telegram_id, ...)` в `try/except: pass`. Текст: `f"Пользователь {name} забронировал слот на {time_str}."`. Диапазон формируется тем же `format_slot_range` (консистентно с остальным ботом), поэтому разделитель — дефис «-», как везде; пример из требования «12:00–12:20» трактуется как тот же диапазон «начало-конец».

_Почему в хендлере, а не в сервисе:_ отправка в Telegram — не доменная логика; сервис остаётся тестируемым без Telegram. Существующее уведомление об отмене уже живёт в хендлере — держим симметрию.

### 5. Админское напоминание — в `ReminderService`, best-effort поверх клиентского

`ReminderService` получает новый параметр конструктора `admin_telegram_id: int` (пробрасывается из `notify.py`, `config.admin_telegram_id`). В `_try_send_reminder` порядок: claim маркера → отправка клиенту (при ошибке unclaim+return, как сейчас) → **затем** отправка админу в отдельном `try/except`, который логирует и НЕ снимает claim. Так клиентская идемпотентность неприкосновенна: админский сбой не вызывает повторной клиентской отправки. Имя клиента — `resolve_client_name(bot, booking.client_id)`. Текст: `f"❗️ Через 5 минут начнётся сессия с пользователем {name}"`.

Асинхронность: сервис синхронный, `send_message`/`get_chat` — корутины. Как и текущий `_send_reminder` (`asyncio.run(result)` при awaitable), админский путь оборачиваем в единую корутину `resolve_client_name(...)`+`send_message(...)` и гоняем через `asyncio.run`; в тестах при синхронных моках awaitable-проверка пропускает `asyncio.run`.

_Альтернатива:_ отдельный маркер `sent_reminders_admin`. Отклонено: изменение схемы/миграции ради best-effort уведомления; потеря админского напоминания при редкой ошибке Telegram допустима (симметрично философии «сбой админ-уведомления не критичен»).

## Risks / Trade-offs

- **Проигравший `Aborted` при гонке «один клиент, два разных слота» получит `SlotTaken`, а не `ClientAlreadyBooked`** → мягкое сообщение о недоступности слота; повторная попытка уже отдаст точную `ClientAlreadyBooked`. Приемлемо, крайне редкий сценарий.
- **Админское напоминание может быть потеряно при ошибке Telegram** (best-effort, без своего маркера) → логируем WARNING; клиентское напоминание при этом гарантированно доставлено. Осознанный трейд-офф против усложнения схемы.
- **Ограничение «одна активная бронь» блокирует запись, если прошлая сессия сегодня не отменена** → соответствует правилу «только на сегодня, один слот»; auto-complete броней вне scope.
- **Лишние `get_chat` на каждый тик напоминаний** → объём мал (только созревшие брони), приемлемо.

## Migration Plan

Изменения аддитивны и обратно совместимы: без схемы БД, конфига и env. Деплой — обычное обновление обеих YC-функций (webhook и notify). Откат — возврат предыдущей версии кода; данные не затрагиваются. Существующие брони и маркеры совместимы.
