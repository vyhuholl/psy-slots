## Context

Доменный слой готов: `bot-config` (длительность, интервалы, TZ, тексты из env), `slot_service` (свободные слоты на сегодня в UTC), `booking_service` (создание/отмена/чтение, race-safe). `bootstrap-bot` дал Dispatcher в webhook-режиме и идемпотентный, stateless-хендлер. Нужен первый пользовательский поток: клиент видит свободные слоты на сегодня, записывается и управляет своими бронями, видя время в своём часовом поясе.

Психолог один и запись только на сегодня, поэтому выбора специалиста и дня нет — сразу список сегодняшних слотов. Два ограничения формируют дизайн: (1) функция эфемерна и не держит состояние между вызовами — значит, никакого серверного FSM; (2) CLAUDE.md требует показывать время в TZ клиента — значит, нужен минимальный профиль клиента с таймзоной.

## Goals / Non-Goals

**Goals:**

- Тонкие aiogram-хендлеры, вся доменная логика — через существующие сервисы.
- Приветствие при `/start` из конфигурации; список сегодняшних свободных слотов → подтверждение, полностью на инлайн-клавиатурах.
- Сообщение о невозможности брони (текст из конфигурации), когда на сегодня слотов нет.
- Навигация без серверного состояния: контекст в `callback_data`.
- Минимальный профиль клиента с таймзоной; запрос при первом контакте; показ времени в этой TZ.
- Создание брони при подтверждении; дружелюбная обработка занятого слота (гонка) и повторного подтверждения.
- Просмотр и soft-отмена своих броней.
- Тестируемость хендлеров без реального Telegram (замоканные сервисы, поддельные `Message`/`CallbackQuery`).

**Non-Goals:**

- Никакой доменной логики броней/слотов заново — только оркестрация существующих сервисов.
- Никакого админ-потока (см. `add-admin-flow`).
- Никаких напоминаний (см. `add-reminder-notifications`) — но профиль клиента здесь готовит для них TZ.
- Никакого выбора специалиста или дня — психолог один, запись только на сегодня.
- Никакого богатого профиля клиента (имя/контакты/история) — только то, что нужно для показа времени.

## Decisions

**Навигация без FSM — состояние в `callback_data`.** Каждая инлайн-кнопка несёт полный контекст шага: `slot:<start_epoch>`, `confirm:<start_epoch>`, `cancel:<booking_id>`, `tz:<name>`. Хендлер восстанавливает выбор из `callback_data` и заново зовёт сервисы — никакого серверного FSM/сессии, что корректно для эфемерной функции. Бюджет `callback_data` — 64 байта: `confirm:` + epoch-seconds (~10) или `cancel:` + UUID (36) укладываются. Альтернатива (aiogram FSM со внешним storage в YDB) добавляет состояние и запись на каждый шаг — избыточно и противоречит stateless-хендлеру.

**Приветствие и «нет слотов» — тексты из конфигурации.** `/start` показывает `WELCOME_MESSAGE` из env. Когда `slot_service` вернул пустой список на сегодня, показываем `BOOKING_UNAVAILABLE_MESSAGE` из env — без ошибки и без пустой клавиатуры. Тексты не хардкодятся в хендлере.

**Профиль клиента — таблица `clients`.** `telegram_id Int64` (PK), `timezone Utf8` (IANA), `created_at Timestamp`. `app/domain/client.py` (валидация TZ через `zoneinfo`) + `app/services/client_service.py` (`get`, `set_timezone`). При первом контакте без TZ бот показывает выбор из нескольких распространённых зон (кнопки `tz:<name>`) и сохраняет выбор; далее TZ берётся из профиля. `telegram_id` совпадает с `client_id` брони и chat_id для будущих напоминаний.

**Дефолт таймзоны.** Если клиент пропустил выбор, используем таймзону конфигурации (env `TIMEZONE`), пока он не задаст свою; сменить можно командой `/timezone`. Так поток не блокируется, но требование «показывать в TZ клиента» выполняется, как только она известна.

**Границы модулей.** `app/bot/handlers/client.py` — тонкие хендлеры команд (`/start`, `/book`, `/mybookings`, `/timezone`) и колбэков; `app/bot/keyboards.py` — сборка клавиатур и единая (де)сериализация `callback_data`; `app/bot/formatting.py` — форматирование UTC→TZ клиента. Хендлеры только: парсят апдейт, зовут сервис, строят ответ. Роутеры подключаются к Dispatcher из `bootstrap-bot`.

**Окно слотов на сегодня.** Сегодняшний день (в таймзоне конфигурации) переводится в UTC-окно `[max(now, day_start), day_end)` и передаётся в `slot_service`; показываются только будущие сегодняшние слоты. Никакого выбора дня — окно всегда «сегодня».

**Обработка занятого слота и повтора.** Подтверждение зовёт `booking_service.create`; на `SlotTaken` хендлер показывает «слот занят, выберите другой» без исключения. Повторное подтверждение (двойной тап/повтор вебхука): защита от двойного бронирования — на стороне `booking-lifecycle`; хендлер при конфликте показывает уже существующую бронь клиента как результат, не создавая вторую.

**Тестирование хендлеров.** Хендлеры вызываются напрямую с поддельными `Message`/`CallbackQuery` и замоканными сервисами; проверяем корректные вызовы сервисов, содержимое клавиатур, приветствие/«нет слотов» и форматирование времени в TZ. Реальный Telegram не нужен — доменная логика уже покрыта в своих changes.

## Risks / Trade-offs

- **Лимит `callback_data` 64 байта.** Mitigation: epoch-seconds/UUID укладываются; единая точка (де)сериализации в `keyboards.py`.
- **Telegram не даёт таймзону пользователя.** Mitigation: явный выбор из списка (+ дефолт из env `TIMEZONE`), команда `/timezone` для смены; сохраняем в профиль.
- **Устаревшие клавиатуры** (клиент жмёт слот из старого сообщения, который уже занят или уже прошёл). Mitigation: `callback_data` несёт `start_utc`, `booking-lifecycle` перепроверяет «сегодня/окно/занятость» при записи; занятый или не-сегодня слот → дружелюбная ошибка.
- **Гонка «двойной тап»** создаёт две попытки записи. Mitigation: идемпотентность вебхука + race-safe guard домена; на один слот проходит одна бронь.
- **DST при показе** (несуществующее/неоднозначное локальное время у клиента). Mitigation: показываем существующие слоты (их даёт `slot_service`), форматирование UTC→TZ клиента однозначно.

## Migration Plan

1. TDD-порядок: сначала профиль клиента (падающие тесты валидации TZ и `set/get`) → домен+сервис+таблица `clients`; затем `callback_data` (де)сериализация и клавиатуры (падающие тесты) → `keyboards.py`; затем форматирование в TZ → `formatting.py`; затем хендлеры (падающие тесты с моками, включая приветствие и «нет слотов») → `handlers/client.py`; подключение роутеров к Dispatcher.
2. Прогнать миграции — создаётся `clients`.
3. Smoke: `/start` → приветствие; `/book` → сегодняшние слоты (или «нет слотов») → подтверждение → бронь; `/mybookings` → отмена.

**Rollback:** снять регистрацию клиентских роутеров (бот перестаёт принимать записи); таблица `clients` идемпотентна и переживает редеплой; данные броней не затрагиваются.
