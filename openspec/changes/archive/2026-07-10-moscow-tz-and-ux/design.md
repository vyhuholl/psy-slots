## Context

Бот обслуживает одного психолога, аудитория — в одном часовом поясе (Москва). Текущая реализация тащит полноценную подсистему таймзон: env-переменная `TIMEZONE`, профиль клиента с IANA-таймзоной (таблица `clients`, `ClientService`, `Client`-домен), команда `/timezone` и клавиатура выбора зон. Это усложняет код и UX без пользы. Параллельно нужно улучшить сам поток: постоянное меню, `/start`, ведущий к записи, диапазонный формат слотов и человекочитаемые имена клиентов у администратора.

Ограничения не меняются: serverless (только webhook, без фонового состояния), test-first, покрытие ≥ 80%, `make validate` без `# type: ignore`.

## Goals / Non-Goals

**Goals:**
- Единая фиксированная таймзона `Europe/Moscow` для трактовки интервалов и показа времени; хранение в БД остаётся в UTC.
- Удалить профиль клиента и всю настройку таймзон (env + пользовательскую) как мёртвый после фиксации домен.
- Постоянная reply-клавиатура с ролевой видимостью админ-кнопки.
- `/start`, диапазонный формат слота, человекочитаемые имена клиента у админа, уведомление админа об отмене клиентом.

**Non-Goals:**
- Деструктивное удаление таблицы `clients` (данные не трогаем; просто перестаём создавать/использовать).
- Хранение имён/username клиента в БД — имена берём вживую из `bot.get_chat`.
- Многочасовые пояса, выбор дня, второй специалист — вне области.

## Decisions

### 1. Таймзона как единая константа `Europe/Moscow`, доступная через `Config.timezone`

Заменяем чтение `TIMEZONE` из env на фиксированную `ZoneInfo("Europe/Moscow")`, но **оставляем свойство `Config.timezone`**, теперь возвращающее эту константу. Тогда `slot_service`, `formatting`, `notify` продолжают читать `config.timezone` — точечное изменение вместо каскада правок.

- *Почему так*: минимальная площадь изменений; единственная точка правды по таймзоне.
- *Альтернатива*: раскидать `ZoneInfo("Europe/Moscow")` по местам показа — отвергнуто (дублирование, риск рассинхрона).
- CLAUDE.md-правило «таймзону не хардкодить» этот change осознанно отменяет (таймзона стала бизнес-правилом, а не конфигом) — CLAUDE.md обновляется в этом же change.

### 2. Удаление профиля клиента

Удаляем `app/domain/client.py`, `app/services/client_service.py`, соответствующие тесты; из `app/migrations.py` убираем `CREATE TABLE clients` (без `DROP` — недеструктивно). `reminder_service` перестаёт зависеть от `ClientService` и форматирует время в `config.timezone` (Москва). Хендлеры `client.py`/`admin.py` больше не зовут `client_service.get(...)`.

- *Почему так*: без таймзоны у профиля не остаётся полей с назначением; сущность мёртвая.
- *Альтернатива*: оставить таблицу и `created_at` — отвергнуто (нет потребителя).

### 3. Reply-клавиатура и маршрутизация по тексту

Reply-кнопки шлют свой текст обычным сообщением. Регистрируем message-хендлеры с фильтром по точному тексту кнопки (`F.text == "✏️ Записаться"` и т.д.), которые вызывают существующие `handle_book` / `handle_my_bookings` / `handle_admin_list`. Клавиатура строится функцией `main_menu_keyboard(is_admin: bool)`; админ-кнопка добавляется только при `is_admin`. Прикрепляем меню к ответу `/start` (как `reply_markup`).

- *Почему так*: reply-меню персонализируется на отправке (per-message `reply_markup`), поэтому ролевую видимость решаем при построении, зная `from_user.id` и `ADMIN_TELEGRAM_ID`.
- *Альтернатива*: inline-клавиатура с `callback_data` — отвергнуто: нужна именно постоянная нижняя reply-клавиатура из ТЗ.

### 4. `/start` = приветствие + слоты, либо сообщение о невозможности

`handle_start` получает `slot_service` и `config`, зовёт `slot_service.list_free_slots()`. Есть слоты → `WELCOME_MESSAGE` + клавиатура слотов; нет → `BOOKING_UNAVAILABLE_MESSAGE`. Меню (reply-клавиатуру) прикрепляем к первому сообщению.

### 5. Диапазонный формат слота

В `slots_keyboard` текст кнопки строим `format_slot_range(slot.start, slot.end, tz)` вместо `format_time(slot.start, tz)`. `callback_data` не меняется (по-прежнему `slot:<epoch>` от `start`).

### 6. Человекочитаемое имя клиента через `bot.get_chat`

Вводим асинхронный хелпер `resolve_client_name(bot, client_id) -> str` (напр. в `app/bot/formatting.py` или новом модуле):
1. `chat = await bot.get_chat(client_id)`;
2. `first/last` → `full = "Имя Фамилия"`; `username` → `@username`;
3. есть имя и username → `f"{full} (@{username})"`; нет имени, есть username → `f"@{username}"`; иначе → `str(client_id)`;
4. любое исключение `bot.get_chat` → `str(client_id)` (деградация).

Так как `admin_bookings_keyboard` синхронна, разрешаем имена **в хендлере** `handle_admin_list`: строим `dict[int, str]` по уникальным `client_id` (дедупликация), передаём в клавиатуру `admin_bookings_keyboard(bookings, tz, names)`. `callback_data` остаётся `admincancel:<id>` — бюджет 64 байта не затрагивается (имя идёт в текст кнопки).

- *Почему в хендлере*: `get_chat` асинхронный и сетевой; клавиатура остаётся чистой sync-функцией, тестируемой на готовом словаре имён.
- *Тот же хелпер* переиспользуется в уведомлении админа об отмене клиентом.

### 7. Уведомление админа об отмене клиентом

`handle_cancel_confirm` после `booking_service.cancel(...)` формирует имя через `resolve_client_name(bot, callback.from_user.id)` и шлёт `ADMIN_TELEGRAM_ID` текст «Пользователь <имя> отменил запись на <диапазон Europe/Moscow>». Отправку оборачиваем в `try/except` (сбой не откатывает отмену), как уже сделано в админском уведомлении клиента.

## Risks / Trade-offs

- **N сетевых вызовов `bot.get_chat` при показе списка админу** → бронирование только на сегодня, записей мало; дедупликация по `client_id` снижает число вызовов.
- **`bot.get_chat` может бросить/таймаутить** → жёсткая деградация до `telegram_id`, поток админа не падает.
- **Удаление `Client`/`ClientService` затрагивает много модулей и тестов** → идём test-first, точечно правим `reminder_service`, `notify`, хендлеры; существующие тесты профиля удаляем вместе с кодом.
- **Отказ от правила «не хардкодить таймзону»** → осознанное решение change; синхронно обновляем CLAUDE.md/README.md, чтобы правило и код не расходились.
- **Reply-меню роутится по тексту** → тексты кнопок — единственная константа маршрутизации; выносим в именованные константы рядом с `main_menu_keyboard`, фильтры хендлеров ссылаются на них.

## Migration Plan

1. Реализация по tasks.md (test-first), `make test` и `make validate` зелёные.
2. Задеплоить обе функции. Убрать `TIMEZONE` из переменных окружения функций.
3. Миграции повторно запускать не обязательно (таблицы не меняются деструктивно); `clients` остаётся, но не используется.
4. Откат: вернуть предыдущую ревизию кода и переменную `TIMEZONE` — данные не менялись, откат безопасен.

## Open Questions

- Прикреплять reply-меню только к `/start` или к каждому ответному сообщению? Базово — к `/start` (и к сообщениям, где логично поддержать навигацию). Достаточно для требований; расширение — при необходимости.
- Где разместить `resolve_client_name`/константы меню — в `formatting.py`/`keyboards.py` или отдельном `app/bot/naming.py`. Решается при реализации; на поведение не влияет.
