## Context

Первый change проекта. Есть только каркас: `pyproject.toml` (aiogram 3.29, ydb 3.30, Python 3.14), пустые `app/` и `tests/`, `Makefile` (`make test`, `make validate`). Целевой рантайм — Yandex Cloud Functions: эфемерный, вызывается по HTTP-событию, между вызовами тёплый инстанс может переиспользоваться, но не гарантированно. Отсюда serverless-ограничения из CLAUDE.md: только webhook, никакого polling, никаких фоновых задач/планировщиков внутри функции, клиент YDB — тёплый на уровне модуля.

Задача change — дать исполняемый и тестируемый каркас: HTTP-событие → aiogram Dispatcher → ответ, плюс переиспользуемый клиент YDB и идемпотентные миграции. Доменной логики (специалисты, расписание, брони) здесь нет.

## Goals / Non-Goals

**Goals:**

- Точка входа YC-функции, разбирающая событие вебхука и прогоняющая апдейт через aiogram в webhook-режиме.
- Валидация секрета вебхука на границе.
- Тёплый модульный клиент YDB + идемпотентный, явно запускаемый механизм миграций.
- Вся конфигурация из env; отсутствие обязательной переменной — явная ошибка.
- Скелет `/start` как end-to-end проверка маршрута.
- Доменная логика тестируется без Telegram — тонкие хендлеры (готовим почву для доменного слоя следующих changes).

**Non-Goals:**

- Никаких доменных сущностей/таблиц (специалист, расписание, бронь) — только механизм миграций, без прикладных таблиц.
- Никакой конкретной регистрации вебхука в Telegram (операционный шаг, не код) и никакого IaC/деплой-скрипта.
- Никакой персистентной дедупликации `update_id` в YDB — идемпотентность на этом этапе обеспечивается тем, что `/start` не имеет побочных эффектов (см. Decisions).

## Decisions

**Разбор события как «сырого» dict, а не через aiohttp-веб-сервер.** YC Functions отдаёт HTTP-событие как dict (`body`, `headers`, ...), а не поднимает сокет. Поэтому не используем `aiogram.webhook.aiohttp_server`; точка входа сама достаёт `body`/`headers`, валидирует секрет, строит `Update.model_validate(...)` и вызывает `dp.feed_update(bot, update)`. Альтернатива (aiohttp app + AsyncServer) не подходит рантайму и добавляет лишний слой.

**Async-мост.** aiogram — async, вход YC-функции — обычная функция. Точка входа синхронна и гоняет корутину через `asyncio.run(...)` (или переиспользуемый loop). Один апдейт на вызов, поэтому короткоживущий `asyncio.run` приемлем и избегает утечки состояния между вызовами.

**Тёплый клиент YDB на уровне модуля.** `ydb.Driver` + `SessionPool`/`QuerySessionPool` создаются при импорте модуля и переиспользуются; `driver.wait(timeout=...)` — лениво при первом обращении. Не закрываем вручную (CLAUDE.md). Конфиг подключения (`YDB_ENDPOINT`, `YDB_DATABASE`, креды) — из env; аутентификация через `ydb.credentials_from_env_variables()` или service-account-совместимый провайдер.

**Конфигурация — один типизированный модуль.** Единая функция/объект читает env один раз и падает с явной ошибкой при отсутствии обязательной переменной (`BOT_TOKEN`, `WEBHOOK_SECRET`, `YDB_ENDPOINT`, `YDB_DATABASE`). Централизует секреты, упрощает мок в тестах. Альтернатива — читать `os.environ` по месту — размазывает валидацию и мешает тестировать.

**Миграции — отдельная явная точка входа, не хендлер.** Идемпотентный `CREATE TABLE IF NOT EXISTS`-модуль, запускаемый как отдельная операция (локальный вызов/деплойный шаг), никогда не на горячем пути вебхука. В этом change прикладных таблиц нет — вводится сам раннер и его каркас; таблицы добавляют доменные changes. Так каждая таблица живёт рядом со своей capability, а миграция остаётся идемпотентной.

**Идемпотентность через отсутствие эффектов, а не дедуп-стор.** Полноценная дедупликация `update_id` требует таблицы и стоит дороже; на этом этапе единственный хендлер `/start` без побочных эффектов, поэтому повтор безопасен по построению. Требование персистентной дедупликации всплывёт с первым мутирующим хендлером (бронь) и будет решаться в том change.

**Границы модулей** (`app/`): `config.py` (env), `ydb_client.py` (тёплый драйвер), `migrations.py` (раннер схемы), `bot.py` (сборка `Bot`/`Dispatcher`, хендлеры), `handler.py` (точка входа YC: разбор события, проверка секрета, мост в aiogram). Разбор/валидация тестируются без сети и без Telegram.

## Risks / Trade-offs

- **Холодный старт + `driver.wait`** → таймаут на первом вызове. Mitigation: разумный таймаут ожидания и понятная ошибка; тёплый инстанс амортизирует.
- **`asyncio.run` на каждый вызов** — небольшой оверхед на создание loop. Mitigation: приемлемо для одного апдейта; при потребности заменяется на переиспользуемый loop, не меняя контракт хендлера.
- **Идемпотентность без дедуп-стора** — держится, только пока хендлеры без побочных эффектов. Mitigation: явно зафиксировано; первый мутирующий change обязан ввести дедуп/защиту от гонок (уже в инвариантах брони).
- **Формат HTTP-события YC** (ключи `body`/`headers`, base64-кодирование) — специфичен для платформы. Mitigation: изолировать разбор события в `handler.py` за узким интерфейсом; тесты гоняют реалистичную форму события.
- **Провайдер кред YDB** различается локально и в YC (SA-токен из метаданных vs env). Mitigation: выбор провайдера в `ydb_client.py` управляется env, тесты мокают драйвер.

## Migration Plan

1. Реализовать модули `app/` (config → ydb_client → migrations → bot → handler) в TDD-порядке: под каждый — сначала падающий тест.
2. Прогнать миграции против serverless-YDB как отдельный шаг (не через функцию).
3. Задеплоить функцию, выставить env (`BOT_TOKEN`, `WEBHOOK_SECRET`, параметры YDB).
4. Зарегистрировать вебхук в Telegram (`setWebhook` с URL функции и `secret_token`) — операционный шаг.
5. Smoke-тест: `/start` возвращает приветствие.

**Rollback:** снять вебхук (`deleteWebhook`) — бот замолкает без потери данных; таблицы идемпотентны и переживают редеплой.

## Open Questions

- Провайдер кред YDB для целевого окружения YC: SA-ключ из env vs токен из метадаты инстанса — уточнить при деплое; на код влияет только выбором провайдера в `ydb_client.py`.
- Нужен ли уже сейчас переиспользуемый event loop вместо `asyncio.run` — отложено до появления замеров латентности.
