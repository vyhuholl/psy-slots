## Context

Психолог один; длительность слота, интервалы доступности (только время суток, без дня недели) и таймзона берутся из конфигурации (env, capability `bot-config`). Бронь — сущность, вокруг которой построены все инварианты CLAUDE.md: персистентная запись с неизменным UUID (а не «слот занят»), явные start/end в UTC с денормализованным end, запрет двойного бронирования с учётом гонок, отмена только soft. Бронь возможна только на сегодня. Целевой рантайм serverless и идемпотентный (webhook может повторно доставить апдейт), поэтому переходы должны быть безопасны к повтору.

Здесь только домен + персистентность брони и единый сервис переходов. Telegram-потоки клиента/администратора — отдельные changes.

## Goals / Non-Goals

**Goals:**

- Таблица YDB `bookings` через существующий раннер миграций.
- Неизменяемая типизированная модель `Booking` + перечисление состояний.
- Единый доменный сервис: создание, отмена, чтение.
- Денормализация `end` при создании; время в UTC.
- Проверка «сегодня + окно доступности + сетка» при создании.
- Race-safe запрет двойного бронирования через сериализуемую транзакцию.
- Идемпотентная soft-отмена, переживание строки.
- Полная тестируемость без Telegram.

**Non-Goals:**

- Никаких aiogram-хендлеров и UX (см. `add-client-booking-flow`, `add-admin-flow`).
- Никакой генерации списка свободных слотов (см. `add-slot-generation`) — здесь только валидация конкретного слота при записи.
- Никакой правки времени брони «на месте»: «изменение» = отмена + новая бронь (см. Decisions).
- Никакой политики DST для локального→UTC — на стороне брони перевод UTC→локальное однозначен; материализация слотов в `add-slot-generation`.

## Decisions

**Схема `bookings`.** `id Utf8` (PK, UUID-строка), `client_id Int64` (telegram user id — он же chat_id для напоминаний), `start_utc Timestamp`, `end_utc Timestamp`, `status Utf8` (`booked`/`cancelled`), `created_at Timestamp`, `cancelled_at Timestamp?` (nullable). Ссылки на специалиста нет — психолог один. Вторичный индекс по `(start_utc)` для проверки пересечений и списков в диапазоне; при необходимости — по `client_id`.

**Состояния — `booked` / `cancelled`; «свободен» = отсутствие активной брони.** Бронь не моделирует «свободный слот» (слоты эфемерны). Состояние хранится строкой; переходы только `booked → cancelled`. Enum в домене (`BookingStatus`).

**`end` денормализуется при создании.** `end_utc = start_utc + timedelta(minutes=D)`, где `D` — длительность из конфигурации на момент создания. Так позднее изменение `D` (env) не сдвигает существующие брони (прямое требование CLAUDE.md). Альтернатива (хранить только start + текущую длительность) нарушила бы это.

**Проверка «сегодня + окно» при записи (defense-in-depth).** Даже если UI покажет слот из `add-slot-generation`, сервис заново валидирует на записи: конвертирует `start_utc` в таймзону конфигурации (`zoneinfo`), проверяет, что дата — сегодняшняя и время ещё не наступило (`SlotNotToday` иначе), берёт минуту суток, ищет интервал доступности, где `interval.start ≤ local_start` и `local_end ≤ interval.end` (`SlotOutsideAvailability` иначе), и проверяет выравнивание `((local_start − interval.start) % D == 0)` (`SlotMisaligned` иначе). Дня недели нет — интервалы применяются к сегодняшней дате. Перевод конкретного UTC-момента в именованную зону однозначен, поэтому DST здесь проблемы не создаёт.

**Race-safe двойное бронирование — сериализуемая интерактивная транзакция YDB.** В одной транзакции: `SELECT` активных броней, пересекающих `[start,end)` (`status='booked' AND start_utc < :end AND end_utc > :start`); если есть — откат с доменной ошибкой «слот занят»; иначе `INSERT` и commit. Оптимистическая сериализуемость YDB отклонит один из двух конкурентных коммитов (Transaction Locks Invalidated) — проигравший трактуется как «слот занят». **Проверка перекрытия, а не равенства start**: если длительность в конфиге менялась, новые слоты иной длины могут пересекать старые брони — overlap это ловит. Альтернатива «UNIQUE(start)» отвергнута: отменённая строка блокировала бы повторную запись, а перекрытия разной длительности не покрываются.

**Отмена — идемпотентный soft-переход.** `booked → cancelled` (+`cancelled_at`); повторная отмена уже отменённой — no-op с успехом (поддерживает идемпотентность вебхука при повторной доставке апдейта). Неизвестный id → доменная ошибка «не найдено». Hard-delete запрещён — строка остаётся читаемой.

**«Изменение» брони = отмена + создание.** Времена брони эффективно неизменны (денормализованный end, immutable id). Перенос реализуется как отмена старой и создание новой брони, а не мутация start/end. Упрощает аудит и согласуется с soft-моделью.

**Границы модулей.** `app/domain/booking.py` — frozen `@dataclass Booking`, `BookingStatus`, ошибки (`SlotNotToday`, `SlotOutsideAvailability`, `SlotMisaligned`, `SlotTaken`, `BookingNotFound`). `app/services/booking_service.py` — читает длительность/TZ/интервалы из конфигурации (`bot-config`); инкапсулирует транзакцию YDB, денормализацию, маппинг строка⇄модель, генерацию `uuid4`. Модель без YDB; сервис без Telegram.

## Risks / Trade-offs

- **TLI/повторы транзакции** под конкуренцией → часть коммитов откатывается. Mitigation: одну ретраибл-ошибку YDB можно повторить; неустранимый конфликт по слоту → «слот занят». Явный контракт: не более одной успешной брони на слот.
- **Стоимость проверки перекрытия** — скан активных броней в окне `[start,end)`. Mitigation: вторичный индекс `(start_utc)`; окно узкое (длина слота).
- **Изменение длительности через env задним числом** может создать теоретическое перекрытие старых и новых слотов. Mitigation: overlap-проверка при создании защищает новые брони.
- **Клиентский дабл-тап** (два апдейта на одну кнопку) → две попытки. Mitigation: идемпотентность вебхука + транзакция; на один слот пройдёт одна. Для разных слотов это легитимно.
- **`client_id` как единственный контакт** — для отображения имени понадобится больше данных. Mitigation: `client_id` достаточно для связи и напоминаний; профиль/контакты — в UI-change при необходимости.
- **DST для локального→UTC** не решается здесь. Mitigation: сторона брони оперирует UTC; политика материализации — в `add-slot-generation`.

## Migration Plan

1. TDD-порядок: падающие тесты модели (денормализация end, состояния) → модель; тесты сервиса (валидация «сегодня»/окна/сетки, вне доступности, перекрытие, повтор после отмены, идемпотентная отмена, чтение) против замоканного пула/транзакции → сервис; затем регистрация таблицы и индекса в раннере миграций.
2. Прогнать миграции против serverless-YDB отдельным шагом — создаются `bookings` и индекс.

**Rollback:** hard-delete не используется; таблица идемпотентна (`IF NOT EXISTS`) и переживает редеплой; откат кода не разрушает данные броней.

## Open Questions

- Нужен ли отдельный индекс по `client_id` сразу или скан по фильтру достаточно на текущих объёмах — по умолчанию индекс `(start_utc)`, `client_id` при росте.
- Политика ретраев сериализуемой транзакции (число повторов, backoff) — начать с одного повтора на ретраибл-ошибку YDB, уточнить по наблюдениям.
- Хранить ли причину/автора отмены (клиент vs администратор) — отложено до UI-changes; сейчас достаточно `cancelled_at`.
