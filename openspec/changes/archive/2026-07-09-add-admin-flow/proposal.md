## Why

Психологу нужно видеть свои записи и отменять их. Настраивать он ничего не может — длительность, интервалы доступности и таймзона берутся из конфигурации (env). Реестра специалистов и привязки к Telegram-записи больше нет, поэтому авторизация админ-действий сводится к сверке с единственным идентификатором администратора из конфигурации. Этот change добавляет тонкий админ-поток поверх существующих read/cancel-запросов сервиса броней.

## What Changes

- Админ-поток на тонких aiogram-хендлерах (навигация без серверного состояния, как в клиентском потоке): просмотр активных броней, отмена брони.
- **Авторизация по конфигурации**: админ-команды доступны только пользователю, чей Telegram-id совпадает с `ADMIN_TELEGRAM_ID` из конфигурации; любой другой пользователь получает отказ и не видит управляющих действий.
- Отмена брони администратором — soft-переход через сервис, с уведомлением клиента.
- Время администратору показывается в таймзоне конфигурации.
- **Нет** настройки длительности/расписания и **нет** реестра/привязки специалиста — эти возможности исключены концепцией.

## Capabilities

### New Capabilities
- `admin-flow`: Telegram-поток администратора — авторизация по `ADMIN_TELEGRAM_ID` из конфигурации, просмотр активных броней и их отмена (soft) с уведомлением клиента, показ времени в таймзоне конфигурации.

## Impact

- **Новый код**: `app/bot/handlers/admin.py` (тонкие хендлеры), расширение `app/bot/keyboards.py` (клавиатуры администратора + `callback_data`), проверка авторизации по `ADMIN_TELEGRAM_ID`, подключение админ-роутеров к Dispatcher.
- **Тесты**: авторизация (совпадение/несовпадение id), просмотр активных броней (в TZ конфигурации, пустой список), отмена брони + уведомление клиента, идемпотентная отмена уже отменённой.
- **Зависит от**: `bot-config` (`ADMIN_TELEGRAM_ID`, TZ), `add-booking-lifecycle` (read-запросы + отмена), `add-client-booking-flow` (общие `app/bot/` утилиты: форматирование, клавиатуры, `callback_data`, chat_id клиента для уведомления), `bootstrap-bot` (Dispatcher).
