## Context

Психолог один; все его параметры (длительность, интервалы доступности, таймзона, тексты) — из конфигурации (env, `bot-config`), настраивать их из бота нельзя. Реестра специалистов и привязки к Telegram-записи нет. Единственная админ-потребность — видеть свои записи и отменять их. Сервис броней (`booking-lifecycle`) уже даёт read-запросы и soft-отмену; здесь — только тонкий Telegram-поток поверх них.

## Goals / Non-Goals

**Goals:**

- Авторизация админ-действий по единственному `ADMIN_TELEGRAM_ID` из конфигурации.
- Просмотр активных броней во времени таймзоны конфигурации.
- Отмена брони (soft) с уведомлением клиента; идемпотентно.
- Тонкие хендлеры без серверного состояния; тестируемость без реального Telegram.

**Non-Goals:**

- Никакой настройки длительности/расписания/таймзоны — всё из env, редактирование исключено концепцией.
- Никакого реестра специалистов и привязки `telegram_id` к записи — психолог один, авторизация по `ADMIN_TELEGRAM_ID`.
- Никакой доменной логики броней заново — только оркестрация `booking_service`.

## Decisions

**Авторизация — сверка с `ADMIN_TELEGRAM_ID`.** Фильтр/мидлварь админ-роутера сравнивает `from_user.id` с `ADMIN_TELEGRAM_ID` из конфигурации; несовпадение → отказ, управляющие действия не показываются. Без реестра и без запросов в БД для авторизации.

**Просмотр — read-запрос активных броней.** Хендлер зовёт read `booking_service` (список активных броней, при желании в горизонте), форматирует время в таймзоне конфигурации через общий `formatting.py`. Пустой список → сообщение без ошибки.

**Отмена — soft через сервис + уведомление клиента.** Кнопка `admincancel:<booking_id>` (с лёгким confirm-шагом) зовёт `booking_service.cancel` (идемпотентный soft-переход), затем `bot.send_message(client_id, …)` уведомляет клиента об отмене во времени клиента. Повторная отмена уже отменённой — no-op без ошибки.

**Границы модулей.** `app/bot/handlers/admin.py` — тонкие хендлеры и колбэки; переиспользует `app/bot/keyboards.py` (клавиатуры/`callback_data`) и `app/bot/formatting.py` из клиентского потока. Роутеры подключаются к Dispatcher из `bootstrap-bot`.

## Risks / Trade-offs

- **`ADMIN_TELEGRAM_ID` не задан / задан неверно** → админ-поток недоступен. Mitigation: валидация переменной в `bot-config`; явная ошибка конфигурации.
- **Уведомление не доставлено** (клиент заблокировал бота) → отмена всё равно фиксируется. Mitigation: отправка в try/except, ошибка логируется, отмена не откатывается.
- **Устаревшая клавиатура** (бронь уже отменена) → повторная отмена. Mitigation: идемпотентность `cancel`.

## Migration Plan

1. TDD-порядок: падающие тесты авторизации (совпадение/несовпадение id) → фильтр; тесты просмотра (список/пусто, TZ) и отмены (+уведомление, идемпотентность) с моками → `handlers/admin.py`; подключение роутеров к Dispatcher.
2. Smoke: админ видит брони → отменяет → клиент получает уведомление; не-админ получает отказ.

**Rollback:** снять регистрацию админ-роутеров (админ-действия недоступны); данные броней не затрагиваются.
