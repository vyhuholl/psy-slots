## Context

`bootstrap-bot` дал тёплый клиент YDB, конфиг из env и идемпотентный раннер миграций, но прикладных таблиц ещё нет. Это первый доменный change: он вводит сущность психолога — корень, к которому привязаны расписание, слоты и брони. По CLAUDE.md два её атрибута редактируемы per-специалист (длительность слота и таймзона), число специалистов — данные, а не код, а все переходы состояния идут через единый доменный сервисный слой, тестируемый без Telegram.

Здесь только домен + персистентность специалиста. Telegram-UI управления специалистом — отдельный change `add-specialist-admin-flow`.

## Goals / Non-Goals

**Goals:**

- Таблица YDB `specialists` через существующий раннер миграций.
- Неизменяемая типизированная доменная модель `Specialist`.
- Доменный сервис `SpecialistService` как единственный вход: register / list / get / update duration / update timezone.
- Валидация: длительность > 0, таймзона — корректный IANA.
- Полная тестируемость без Telegram и без сети (YDB замокан/локален).

**Non-Goals:**

- Никаких aiogram-хендлеров и UX (см. `add-specialist-admin-flow`).
- Никакого недельного расписания и слотов (`add-weekly-availability`, `add-slot-generation`).
- Никакого разрешения конфликта «изменение длительности vs существующие брони» — броней ещё нет; инвариант решается в change, где сходятся длительность и бронь.
- Никакого удаления специалистов (вне области; при необходимости — отдельный soft-подход позже).

## Decisions

**Схема `specialists`.** Колонки: `id Utf8` (PK, UUID-строка), `name Utf8`, `slot_duration_minutes Uint32`, `timezone Utf8` (IANA-имя), `created_at Timestamp`. PK — строковый UUID. YDB не имеет нативного UUID-типа, поэтому храним каноническую строку `uuid4`. Альтернатива (`Uint64`/случайное число) хуже читается и рискует коллизиями формата с брони, где UUID — явное требование.

**Длительность — `Uint32` в минутах.** Простое целое, покрывает «~15 мин», легко валидируется (> 0). Позднее в `add-booking-lifecycle` `end` считается как `start + timedelta(minutes=duration)`. Альтернатива — YDB `Interval` — избыточна и неудобна для правки человеком.

**Таймзона — IANA-строка, валидируется `zoneinfo.ZoneInfo`.** Храним имя (`Europe/Moscow`), а не смещение: смещение «плывёт» на переходах DST, а расписание задаётся в именованной зоне специалиста. Валидация — попытка сконструировать `ZoneInfo(name)`; неуспех → доменная ошибка. Стандартная библиотека, без внешней зависимости.

**Границы модулей.** `app/domain/specialist.py` — frozen `@dataclass Specialist` + доменные ошибки (`SpecialistNotFound`, `ValidationError`); валидация в конструкторе/фабрике модели. `app/services/specialist_service.py` — сервис над YDB (принимает драйвер/пул из `ydb_client`), маппинг строка⇄модель, генерация UUID при регистрации. Модель не знает про YDB; сервис не знает про Telegram. Так домен-валидация тестируется чисто, а сервис — против замоканного пула.

**«Не найдено» — явный результат, не исключение хранилища.** `get` возвращает `Specialist | None` (или `Optional`); мутации несуществующего id поднимают доменную `SpecialistNotFound`. Низкоуровневые ошибки YDB не протекают в вызывающий код. Явный контракт проще тестировать и использовать в будущих хендлерах.

**Регистрация специалистов — операция сервиса.** Наполнение реестра (сейчас двое) делается через `register(...)` из сид/админ-скрипта, не через пользовательский поток. UI-редактирование придёт в admin-flow поверх этого же сервиса.

**Миграция идемпотентна и растёт по месту.** Добавляем `CREATE TABLE IF NOT EXISTS specialists ...` в раннер из `bootstrap-bot`. Каждая capability добавляет свои таблицы в тот же идемпотентный раннер — так таблица живёт рядом со своим доменом, а повторный прогон безопасен.

## Risks / Trade-offs

- **UUID как `Utf8`** — теряем типовую строгость БД. Mitigation: единый маппинг в сервисе, генерация только `uuid4`, формат покрыт тестом.
- **Last-write-wins при правке** длительности/таймзоны — параллельные правки одной записи могут затирать друг друга. Mitigation: специалист правит свою запись, конкуренция маловероятна; при необходимости позже — версия/optimistic concurrency. Для конфиг-атрибутов приемлемо.
- **Изменение длительности задним числом** повлияет на будущую генерацию слотов, но не на существующие брони (там `end` денормализован — уже заложено в модель брони). Здесь броней ещё нет, поэтому конфликт вне области; фиксируем связь для `add-booking-lifecycle`.
- **Валидность IANA зависит от tzdata ОС/окружения.** Mitigation: полагаемся на `zoneinfo` + системную/`tzdata`-базу; при пустой базе в окружении — добавить пакет `tzdata` (отслеживается при деплое).

## Migration Plan

1. TDD-порядок: сначала падающие тесты модели (валидация длительности/таймзоны), затем модель; затем тесты сервиса (register/list/get/update) против замоканного пула, затем сервис; затем регистрация таблицы в раннере миграций.
2. Прогнать обновлённые миграции против serverless-YDB отдельным шагом (не через функцию) — создаётся таблица `specialists`.
3. Наполнить реестр через `register(...)` (сид) для текущих специалистов.

**Rollback:** таблица `specialists` пуста/не используется другими change до их реализации; откат кода не требует разрушения таблицы (идемпотентный `IF NOT EXISTS` переживает редеплой).

## Open Questions

- Хранить ли `created_at`/`updated_at` для аудита правок — включаю `created_at`; `updated_at` отложен до появления реальной потребности.
