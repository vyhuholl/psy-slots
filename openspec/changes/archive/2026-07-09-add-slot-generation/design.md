## Context

Расписание (`weekly-availability`) хранит абстрактные окна в локальном времени специалиста, длительность живёт на записи специалиста, а брони (`booking-lifecycle`) заняли часть времени. Клиентскому потоку нужен конкретный список свободных слотов в UTC. По CLAUDE.md слоты эфемерны — генерируются из расписания с шагом длительности и нигде не хранятся, а инвариант «слот только внутри окна доступности» здесь и материализуется.

Это change — единственное место, где абстрактное локальное расписание превращается в конкретные UTC-моменты, поэтому именно тут решается политика DST (перевод локального→UTC), которую предыдущие changes сознательно отложили.

## Goals / Non-Goals

**Goals:**

- Чистая, детерминированная генерация слотов: шаг = длительность, без хвостового слота за концом интервала.
- Материализация локального времени интервалов в UTC для дат окна.
- Явная политика DST: пропуск несуществующего локального времени, дедуп неоднозначного (`fold=0`).
- Исключение слотов, пересекающихся с активными бронями.
- Запрос по окну `[from, to)` в UTC, упорядоченная выдача.
- Разделение чистого ядра (без I/O, полностью юнит-тестируемого) и сервиса-оркестратора.

**Non-Goals:**

- Никакой персистентности слотов (эфемерны; таблиц и миграций нет).
- Никаких aiogram-хендлеров и форматирования для показа (см. `add-client-booking-flow`).
- Никакого создания/отмены броней (см. `add-booking-lifecycle`) — здесь только чтение активных для исключения.
- Никакой гарантии, что показанный свободный слот останется свободным к моменту записи — это совещательная выдача; финальную защиту даёт `add-booking-lifecycle` при записи.
- Никаких интервалов через полночь (расписание их не поддерживает).

## Decisions

**Чистое ядро отдельно от сервиса.** `app/domain/slots.py` — frozen `Slot(specialist_id, start_utc, end_utc)` и чистая функция `generate(intervals, duration, tz, window) -> list[Slot]` без обращения к YDB: вся математика сетки и DST здесь, покрывается юнит-тестами без сети. `app/services/slot_service.py` — оркестратор: тянет специалиста (длительность+TZ), интервалы доступности и активные брони, зовёт ядро, вычитает занятые. Так самая хитрая часть (DST) тестируется чисто.

**Сетка слотов — полуоткрытая, без хвоста.** Внутри интервала `[start_min, end_min)` слоты идут `t = start_min, start_min+D, ...`, пока `t + D ≤ end_min`. Слот, выходящий за конец, не порождается. Совпадает с проверкой выравнивания в `add-booking-lifecycle`, чтобы показанный слот проходил валидацию при записи.

**Перебор дат по окну в TZ специалиста.** Для окна `[from_utc, to_utc)` перебираем локальные даты специалиста, покрывающие окно (с запасом ±1 день на краях из-за смещения зоны), для каждой даты с подходящим `weekday` материализуем слоты и в конце фильтруем по `from_utc ≤ start_utc < to_utc`. Запас снимает off-by-one на границах суток/зоны.

**Политика DST — пропуск несуществующего, `fold=0` для неоднозначного.** Локальное `datetime` строим с `tzinfo=ZoneInfo(tz)`. Проверка существования — round-trip: перевести в UTC и обратно в локаль; если настенное время не совпало с исходным, локального момента не существует (весенний переход) → слот пропускаем. Для неоднозначного часа (осенний переход) берём `fold=0` — ровно одно детерминированное вхождение, без дубля. Стандартная библиотека `zoneinfo`, без внешних зависимостей. Альтернатива (наивно доверять первому переводу) давала бы «сдвинутые» или задвоенные слоты на переходах.

**Исключение занятых через overlap с активными бронями.** Тянем `list_active_for_specialist(from, to)` из `booking-lifecycle` и выкидываем слот, если его `[start,end)` пересекается с любой активной бронью (`a.start < b.end and b.start < a.end`). Отменённые не в выдаче активных, поэтому их слоты снова свободны — требование «отменённая освобождает слот» выполняется автоматически.

**Совещательный характер выдачи.** Между генерацией и записью бронь может появиться (другой клиент, гонка). Не пытаемся это исключить на уровне слотов — источник истины и защита от двойного бронирования в `add-booking-lifecycle`. Так генерация остаётся чистой read-моделью без блокировок.

## Risks / Trade-offs

- **Корректность DST** — самая тонкая часть. Mitigation: чистое ядро + прицельные тесты на оба перехода (весна/осень) для реальной зоны специалиста; политика зафиксирована (skip / fold=0).
- **Off-by-one на границах окна/суток** из-за смещения зоны. Mitigation: перебор дат с запасом ±1 день и финальный фильтр по UTC-окну.
- **Гонка «показали свободным — заняли к записи»**. Mitigation: осознанно; повторная валидация и race-safe guard при записи в `add-booking-lifecycle`; UI покажет ошибку «слот занят».
- **Стоимость генерации** на большом окне × много интервалов. Mitigation: окно ограничивает вызывающий (напр. 14 дней); брони тянутся одним запросом по окну.
- **Зависимость валидности TZ от tzdata окружения.** Mitigation: та же, что в реестре — полагаемся на `zoneinfo`/`tzdata`; отслеживается при деплое.

## Migration Plan

1. TDD-порядок: падающие тесты чистого ядра (шаг/границы без хвоста, материализация в UTC, оба DST-случая) → ядро `slots.py`; затем тесты сервиса (исключение занятых, отменённая освобождает, фильтр по окну, пустая выдача) с замоканными репозиториями → `slot_service.py`.
2. Персистентности нет — миграции не затрагиваются, деплой не требует изменений схемы.

**Rollback:** чистая read-модель без побочных эффектов; откат кода ничего не разрушает.
