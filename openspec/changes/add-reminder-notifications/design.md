## Context

Продукт работает: клиент записывается, специалист управляет расписанием. Не хватает напоминаний. Serverless-ограничения из CLAUDE.md прямо запрещают планировщики и фоновые задачи внутри функции, поэтому напоминания — отдельная функция по timer-триггеру YC, которая на каждый тик находит «созревшие» напоминания и рассылает их. Функция переиспользует тёплый клиент YDB, конфиг и раннер миграций из `bootstrap-bot`, read-запросы броней из `booking-lifecycle` и профиль клиента (TZ, chat_id) из `client-profile`.

## Goals / Non-Goals

**Goals:**

- Отдельная точка входа для timer-триггера, без polling и без планировщиков внутри.
- Идемпотентная рассылка: персистентный маркер отправки по (бронь, вид смещения).
- Настраиваемые смещения из env; произвольное их число.
- Только активные будущие брони; отменённые/прошедшие — мимо.
- Текст во времени клиента.
- Устойчивость к сбоям отправки; тестируемость без реального Telegram.

**Non-Goals:**

- Никаких изменений webhook-функции и доменных сервисов — только чтение броней и отправка.
- Никакого планировщика в коде — расписание тиков задаётся timer-триггером YC (операционно).
- Никаких напоминаний специалисту в этом объёме — опционально/конфигурируемо, вынесено в Open Questions.
- Никакой отмены/переноса из напоминания — только информирование (кнопки действий можно добавить позже).

## Decisions

**Отдельная точка входа `app/notify.py`.** Обработчик timer-триггера YC, независимый от `handler.py` (webhook). Разворачивается как вторая YC-функция с тем же кодом/образом, но своим entrypoint и cron-триггером. Внутри — один проход: найти созревшие напоминания, разослать, завершиться. Никаких `asyncio`-фоновых задач; при необходимости async-отправки — короткоживущий `asyncio.run`, как в webhook.

**Логика «созрело и не отправлено», а не точное окно тика.** Для каждого смещения `X` напоминание по брони созревает, когда `now ≥ start − X` и `start > now` (ещё не началась) и маркер `(booking_id, X)` отсутствует. Это устойчиво к пропущенным тикам и дрожанию таймера: даже если тик был пропущен, следующий разошлёт всё созревшее. Альтернатива (слать только попавшим ровно в `[start−X, start−X+tick)`) хрупка к джиттеру и пропускам.

**Идемпотентность — таблица `sent_reminders` с claim-then-send.** Схема: PK `(booking_id Utf8, kind Utf8)`, `sent_at Timestamp`. Перед отправкой делаем условную вставку-claim маркера (insert-if-not-exists по PK); если маркер уже есть — пропускаем (дубль). После успешной отправки claim остаётся; при ошибке отправки claim удаляется, чтобы напоминание повторилось на следующем тике. `kind` — строковый код смещения (напр. `24h`, `1h`). Так исключены дубли даже при перекрывающихся тиках (PK-гонку выигрывает один), а сбой отправки не «съедает» напоминание. Альтернатива (флаг на брони) не масштабируется на несколько смещений.

**Смещения из конфигурации.** `REMINDER_OFFSETS` в env (напр. `24h,1h` или минуты) парсится в `config.py` в список длительностей; число произвольно, ничего не захардкожено. `kind` каждого напоминания выводится из смещения.

**Выборка кандидатов.** На тик берём активные будущие брони в горизонте `[now, now + max(offsets)]` через `booking_service.list_active_for_specialist`/эквивалентный запрос по всем специалистам (или по индексу `(start_utc)`), затем для каждой брони × каждое смещение проверяем зрелость и маркер. Горизонт ограничен максимальным смещением — выборка мала.

**Отправка и формат.** Тёплый `Bot` на уровне модуля (токен из env), `bot.send_message(client_id, text)`; `client_id` = chat_id. Время — через переиспользуемый `formatting.py` в TZ клиента из `client-profile`. Дефолтная TZ, если у клиента не задана (как в клиентском потоке).

**Устойчивость.** Каждое напоминание — в try/except: ошибка одного получателя логируется, claim снимается для повтора, цикл продолжается; вызов функции не падает. Ретраи в пределах окна ограничены тем, что после `start` бронь перестаёт быть будущей и выпадает из кандидатов.

## Risks / Trade-offs

- **Двойная отправка при сбое между claim и записью успеха** практически исключена (claim до отправки; повтор только при явной ошибке отправки). Редкий дубль при аварии инстанса после отправки, но до фиксации — приемлемее пропуска. Mitigation: claim-then-send; зафиксировано.
- **Клиент заблокировал бота** → отправка всегда падает, напоминание ретраится каждый тик до `start`. Mitigation: ограничение горизонтом окна; при необходимости — счётчик попыток/бэкофф (точка расширения).
- **Пропущенные тики / простой функции** → задержка, но не потеря: «созрело и не отправлено» разошлёт на следующем тике (возможно позже желаемого). Mitigation: осознанный компромис; для «за 1 час» держать частоту тиков достаточной.
- **Выборка по всем специалистам** без узкого индекса по `start_utc`. Mitigation: горизонт мал; при росте — индекс/материализованный запрос по времени начала.
- **Смещение/TZ отображения** — перевод UTC→TZ клиента однозначен; DST не мешает. Mitigation: та же `formatting.py`.

## Migration Plan

1. TDD-порядок: падающие тесты сервиса (зрелость по смещению, идемпотентность маркера, пропуск отменённых/прошедших, формат в TZ, устойчивость к ошибке) → `reminder_service.py`; тест точки входа (тик обрабатывает и завершается) → `notify.py`; парсинг `REMINDER_OFFSETS` → `config.py`; регистрация таблицы → `migrations.py`.
2. Прогнать миграции — создаётся `sent_reminders`.
3. Задеплоить вторую YC-функцию с entrypoint `notify` и timer-триггером (cron); те же env-секреты.
4. Smoke: создать бронь в пределах смещения → на тике клиент получает напоминание; повторный тик дубль не шлёт.

**Rollback:** отключить timer-триггер (рассылка останавливается, данные целы); таблица `sent_reminders` идемпотентна и переживает редеплой; webhook-функция не затрагивается.

## Open Questions

- Частота timer-триггера (cron) — должна быть не грубее наименьшего смещения (для «за 1 час» — не реже раза в ~10–15 мин). Согласовать при деплое.
- Напоминать ли также специалиста и с какими смещениями — по умолчанию нет; включаемо конфигом.
- Нужен ли счётчик попыток/бэкофф для заблокировавших бота клиентов — по умолчанию нет, ретрай в пределах окна.
- Единый запрос активных будущих броней по времени (по всем специалистам) vs перебор специалистов — уточнить с учётом индексов при росте.
